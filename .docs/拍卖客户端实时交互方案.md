# 拍卖客户端实时交互方案

## 1. 目标

客户端需要配合后端的 Redis Lua 裁决、WebSocket Gateway 推送和快照恢复机制，保证用户在高并发和网络抖动下仍能看到正确的当前价、领先者、倒计时和成交结果。

客户端原则：

- 出价最终结果以后端返回和 WebSocket 事件为准。
- 本地倒计时只用于展示，不能自行判定竞拍最终结束。
- WebSocket 消息可能重复、乱序、丢失，客户端必须使用 `seq` 去重，并支持快照恢复。
- 前端防抖/节流只优化体验，不能替代后端限流和裁决。

## 2. 客户端整体链路

推荐链路：

```text
进入直播间/拍品页
  -> 调 HTTP 接口获取拍品/直播间基础信息
  -> 建立 WebSocket 连接
  -> 接收服务端 snapshot
  -> 使用 snapshot 初始化当前价、领先者、endTime、seq
  -> 持续接收增量事件
  -> 用户出价走 HTTP API 或约定出价接口
  -> 出价结果以后端返回和后续事件修正为准
```

如果 WebSocket 断开：

```text
断线
  -> UI 显示连接恢复中
  -> 指数退避重连
  -> 重连成功后重新拉 snapshot
  -> 用 snapshot 修正页面状态
  -> 继续接收增量事件
```

## 3. 连接初始化

客户端进入拍卖页后应完成：

1. 获取登录态和 access token。
2. 拉取直播间和拍品基础信息。
3. 建立 WebSocket：

```text
wss://realtime.example.com/ws/live-rooms/{roomId}
或
wss://realtime.example.com/ws/auctions/{auctionId}
```

4. 连接成功后等待服务端 snapshot：

```json
{
  "type": "snapshot",
  "auctionId": 10001,
  "status": "RUNNING",
  "currentPrice": 12000,
  "leaderBidderId": "u_1001",
  "endTime": "2026-05-25T10:00:00Z",
  "seq": 18,
  "serverTime": "2026-05-25T09:59:30Z"
}
```

5. 使用 snapshot 初始化本地状态：

- `currentPrice`
- `leaderBidderId`
- `status`
- `endTime`
- `lastSeq`
- `serverTimeOffset`

`serverTimeOffset` 用于修正本地倒计时：

```text
serverTimeOffset = serverTime - clientNow
remainMs = endTime - (clientNow + serverTimeOffset)
```

客户端不要使用“收到 snapshot 的本地时间”作为倒计时起点，也不要使用设备本地绝对时间直接判断结束。不同用户设备时间可能不一致，必须通过 `serverTime` 修正。

## 4. WebSocket 心跳与重连

客户端需要支持心跳：

- 收到 server `ping` 后及时返回 `pong`。
- 如果协议层由浏览器自动处理 ping/pong，则业务层仍建议监听长时间无消息。
- 超过 2 - 3 个心跳周期未收到服务端消息，主动断开并重连。

重连策略：

```text
第 1 次：1s 后重连
第 2 次：2s 后重连
第 3 次：5s 后重连
之后：10s 或 15s 间隔重连
```

要求：

- 页面不可重复创建多个 WebSocket 连接。
- 用户切换直播间/拍品时，先关闭旧连接再建立新连接。
- 重连成功后必须重新拉 snapshot，不能只依赖断线前本地状态。
- 重连期间出价按钮可以禁用，或提示“连接恢复中，请稍后重试”。

## 5. 事件处理与 seq 去重

所有服务端事件必须带 `seq`。客户端维护 `lastSeq`：

```text
event.seq <= lastSeq -> 丢弃
event.seq > lastSeq  -> 应用事件，并更新 lastSeq
```

常见事件：

```json
{
  "type": "bid.accepted",
  "auctionId": 10001,
  "currentPrice": 13000,
  "leaderBidderId": "u_1002",
  "endTime": "2026-05-25T10:00:30Z",
  "extendCount": 1,
  "seq": 19
}
```

```json
{
  "type": "bid.rejected",
  "auctionId": 10001,
  "reason": "BELOW_MIN_INCREMENT",
  "currentPrice": 13000,
  "seq": 20
}
```

```json
{
  "type": "auction.closed",
  "auctionId": 10001,
  "status": "CLOSED_WON",
  "winnerId": "u_1002",
  "price": 13000,
  "orderId": 90001,
  "seq": 25
}
```

不同事件处理：

- `snapshot`：直接覆盖本地实时状态，并设置 `lastSeq`。
- `bid.accepted`：更新当前价、领先者、`endTime`、`serverTimeOffset`、延时次数。
- `bid.rejected`：如果是当前用户发起的请求，展示失败原因；不应回滚服务端当前价。
- `timer.extended`：更新 `endTime` 和 `serverTimeOffset`。
- `ranking.updated`：更新排行榜；可节流渲染。
- `auction.closed`：设置终态，停止倒计时，禁用出价按钮。

## 6. 出价交互

客户端出价前可以做基础校验：

- 价格必须大于 0。
- 价格必须高于页面当前价。
- 价格要满足页面展示的最小加价规则。
- 未登录、未报名、保证金未 ready 时禁用出价。
- 本地按钮节流，例如 500ms 内只允许点击一次。

但这些只能优化体验。最终出价是否有效以后端 Redis Lua 裁决结果为准。

出价请求需要携带幂等键：

```text
Idempotency-Key: uuid
```

同一次用户点击只生成一个幂等键。请求超时后如果用户点击重试，应复用同一个幂等键，直到得到明确结果或用户重新编辑价格。

推荐交互：

```text
点击出价
  -> 按钮进入 submitting
  -> 发送请求
  -> 收到 accepted：更新本地“出价已提交/领先中”
  -> 收到 rejected：展示失败原因
  -> 请求超时：提示“确认中”，等待 WebSocket 事件或允许使用同幂等键重试
```

不要在请求超时时立即判定出价失败。服务端可能已经裁决成功，只是响应丢失。

## 7. 最后一秒出价与结束确认

客户端倒计时到 0 时，不能直接展示“已结束”。必须进入结束确认阶段：

```text
倒计时到 0
  -> UI 显示“结束确认中”
  -> 等待约 1000ms
  -> 如果收到 bid.accepted / timer.extended：
       使用事件中的 endTime 重置倒计时，继续竞拍
     如果收到 auction.closed：
       展示成交/流拍结果
     如果 1000ms 后没有收到事件：
       调接口拉取当前拍品状态或重新请求 snapshot
       以服务端状态为准
```

原因：

- 最后一秒有效出价可能触发服务端防狙击延时。
- WebSocket 广播可能有几十到几百毫秒延迟。
- 客户端本地时间和服务端时间可能有偏差。

## 8. UI 状态建议

建议明确区分这些状态：

- `connecting`：正在连接 WebSocket。
- `live`：连接正常，拍品运行中。
- `submittingBid`：当前用户出价请求提交中。
- `confirmingEnd`：倒计时归零后等待服务端确认。
- `reconnecting`：连接断开，正在重连。
- `closedWon`：已成交。
- `closedFailed`：已流拍。
- `error`：无法恢复的错误。

出价按钮禁用条件：

- 未登录。
- 不是买家角色。
- 未报名或保证金未 ready。
- WebSocket 未连接且无法确认当前状态。
- 拍品状态不是 `RUNNING` / `EXTENDED`。
- 当前正在提交出价。
- 当前处于 `confirmingEnd`。
- 拍品已终态。

## 9. 排名、统计和倒计时渲染

为避免高频消息导致页面卡顿：

- 排名更新可 100ms - 300ms 节流渲染。
- 在线人数、围观人数可 1s 级别更新。
- 倒计时使用本地 `requestAnimationFrame` 或定时器计算，不要求服务端每秒推 `timer.tick`。
- 倒计时公式固定为 `remainMs = endTime - (Date.now() + serverTimeOffset)`。
- 服务端推送新的 `endTime` / `serverTime` 时，客户端更新 `serverTimeOffset` 并修正倒计时。
- 当页面进入后台、系统休眠或定时器被浏览器降频后，页面恢复时应立即按当前 `Date.now()` 重新计算倒计时，不要累计旧 tick。
- 倒计时展示可以按 100ms 或 1s 粒度渲染，但结束判定仍要进入 `confirmingEnd`，等待服务端事件或拉取 snapshot。
- 长列表排行榜使用虚拟列表或只展示 Top N。

## 10. 错误提示映射

客户端应对常见拒绝原因给出明确提示：

| reason | 建议提示 |
| --- | --- |
| `INVALID_STATE` | 当前拍品不可出价 |
| `BLACKLIST` | 当前账号暂不可参与竞拍 |
| `NOT_ENROLLED` | 请先报名 |
| `DEPOSIT_NOT_READY` | 保证金未完成 |
| `FREQ_LIMIT` | 出价过于频繁，请稍后再试 |
| `BELOW_MIN_INCREMENT` | 出价未达到最低加价幅度 |
| `AUCTION_CLOSED` | 拍卖已结束 |

错误提示不应阻塞 WebSocket 事件继续更新页面。

## 11. 客户端缓存

客户端可以缓存展示型数据：

- 商品标题、图片、描述。
- 直播间标题、封面。
- 商家基础信息。
- 已结束拍卖结果。

不能缓存作为裁决依据的数据：

- 当前最高价。
- 领先者。
- 拍品状态。
- 结束时间。
- 保证金 ready 状态。

这些实时数据必须以 snapshot 和 WebSocket 增量事件为准。

## 12. 验收清单

- 断网后能自动重连，并用 snapshot 修正状态。
- 重复事件不会导致价格回退。
- 乱序事件不会覆盖新状态。
- 最后一秒出价后客户端不会提前展示结束。
- 请求超时后不会错误提示“出价失败”，能通过 WebSocket 或重试确认结果。
- 慢网络下页面不会卡死，排名和统计有节流渲染。
- 切换直播间不会残留旧 WebSocket 连接。
- 拍品终态后出价按钮禁用。
