# 高并发与压测技术报告（v2 异步裁决版）

> 本报告面向实时拍卖直播间后端的出价主链路，重点复盘从同步裁决到异步裁决的架构演进与压测结果。全文按 4 次方案演进展开：v1 同步竞价 baseline → v2 同步链路优化（异步落库、排行榜异步、Lua 精简）→ v3 异步裁决（单拍品绑定单 Kafka 分区与单工作线程）→ v4 异步裁决（16 分区随机分发，并由 Redis 同分片 Lua 保证同拍品顺序性）。v3/v4 数据来自本轮三段实测窗口（Stage A/B/C）的监控聚合结果；v1/v2 数字沿用前序同步链路压测沉淀。

## 0. 修订说明

本次补充 **异步裁决链路（v3 / v4）** 压测结论，并对原同步链路（v1 / v2）内容做摘要化保留。报告定位从“代码实现说明”调整为“项目架构与压测复盘”：正文不再展开具体文件路径、内部变量名或 Prometheus 原始指标名，只保留读者理解架构、瓶颈和数据结论所需的信息。

- **v3（出价链路异步化第一版）**：出价 ACK（Acknowledgement，客户端收到的入队确认）与后端裁决解耦。ACK 只表示“请求已进入裁决队列”，真正的出价裁决由 Kafka 消费侧异步完成。为了保证同一个拍品的出价顺序，v3 采用 **1 个拍品 → 1 个 Kafka 分区 → 1 个 worker（工作线程）串行消费**。Stage A 实测窗口（2026-06-09 05:51:58 ~ 05:57:01，2 个拍品 × 500 QPS）暴露出单分区、单 worker 的串行天花板：ACK QPS 能稳定在 1000 左右，但**裁决吞吐仅约 482/s，约半数请求因队列背压被拒绝（Queue reject，入队拒绝，约 497/s）**，Result（最终裁决结果）延迟 P95（第 95 百分位延迟）升至 **3.04s**。
- **v4（异步裁决最终版）**：把“Kafka 分区数”和“同拍品顺序性”解耦。Kafka 侧改为 **16 分区随机分发**，让 worker pool（工作线程池）并行消费；同拍品顺序性下沉到 **Redis 同分片 Lua 单线程执行**来兜底。Stage B（500 QPS/拍品 × 2 个拍品 = 1000 总 QPS）下，ACK P95 = 25ms、Result P95 = 107ms、Kafka lag（Kafka 待消费消息积压量）= 0、Queue reject = 0。Stage C（1000 QPS/拍品 × 2 个拍品 = 2000 总 QPS）下，ACK P95 仍只有 23ms，但 worker pool 已接近极限：Worker inflight（工作池在途任务数）= 32/32、Kafka lag 平均 227 / 峰值 413、Queue reject 约 630/s，说明系统已逼近当前架构的吞吐拐点。

原同步链路 5 次压测（T1–T5）的关键数字和结论保留在 2.1 / 2.2 节，用于与 v3/v4 横向对比。

> **数据可追溯性**：本报告中的 v3/v4 数字均来自对应实测窗口的监控系统聚合；同步链路（v1/v2）数字沿用原报告 T1–T5 实测结果。本次修订不重新拉取 Prometheus 数据，也不调整任何压测数字。

## 1. 测试方法与环境

### 1.1 工具与并发模型

- **压测方式**：使用本地压测控制台和多虚拟买家压测引擎，通过 HTTP 控制测试生命周期，并实时回看压测指标。
- **并发模型**：每个虚拟买家保持一条 WebSocket 连接，并拥有独立发送循环和出价任务队列；调度器按目标 QPS 自适应分发请求。
- **价格生成**：压测端严格读取后端返回的加价规则和当前期望价格，避免因压测端自造无效价格而产生虚假的“低于最小加价”拒绝。
- **部署环境**：8 核 16G 单机，Redis、Kafka 与服务进程同台部署。

### 1.2 监控口径与指标定义

| 指标 | 含义 | 观察口径 |
| --- | --- | --- |
| ACK QPS | 出价 ACK 速率。v3/v4 中 ACK 只表示“入队成功”，不代表裁决完成。 | 单位时间内客户端收到入队确认的数量。 |
| ACK 延迟 | 客户端发起出价到收到 ACK 的耗时。 | 以 P50 / P95 / P99 等分位数观察。P50 是中位延迟，P95/P99 分别表示 95%/99% 请求不超过该延迟。 |
| Result 延迟 | 客户端发起出价到收到最终裁决结果的端到端耗时。 | 反映用户真正知道“成交 / 失败”的等待时间。 |
| Decision 耗时 | worker 实际执行一次裁决的耗时，主要包含 Redis Lua 执行与网络往返。 | 用于判断瓶颈在裁决执行本身，还是在排队等待。 |
| 裁决排队 | 进入 Kafka 前的服务内存队列深度。 | 队列接近容量上限时，说明背压即将或已经出现。 |
| Kafka lag | Kafka 待消费消息积压量。 | lag 为 0 表示消费能力跟得上生产；持续升高表示裁决消费侧跟不上入口流量。 |
| Worker inflight | worker pool 中正在处理的在途任务数。 | 接近或达到线程池容量时，说明裁决消费侧已经打满。 |
| 进程 CPU | 进程级 CPU 使用量，多核累加。 | 用于判断是否为 CPU 饱和。 |
| Goroutines | Go 运行时协程数量。 | 用于观察并发资源水位。 |
| RSS | 进程驻留内存。 | 用于观察内存水位。 |
| Queue reject | 入队拒绝速率。通常由内存队列满或 Kafka 写入失败触发，客户端在 ACK 阶段直接收到拒绝。 | 是背压已经外显到用户侧的信号。 |
| 裁决结果速率 | 单位时间内裁决为成交（accepted）或拒绝（rejected）的数量。 | accepted 表示出价被接受，rejected 表示出价被业务规则拒绝。 |

> **三段实测窗口（异步链路 v3/v4）**：
>
> | Stage | 链路 | 窗口（UTC） | 拓扑 | QPS 模型 |
> | --- | --- | --- | --- | --- |
> | A | v3 | 2026-06-09 05:51:58.843 ~ 05:57:01.282 | 1 个拍品绑定 1 个分区和 1 个 worker | 2 个拍品 × 500 = 1000 总 QPS |
> | B | v4 | 2026-06-09 07:50:35.760 ~ 08:03:47.556 | 16 分区随机分发 + Redis 同分片 Lua 保序 | 2 个拍品 × 500 = 1000 总 QPS |
> | C | v4 | 2026-06-09 08:08:24.801 ~ 08:13:07.404 | 16 分区随机分发 + Redis 同分片 Lua 保序 | 2 个拍品 × 1000 = 2000 总 QPS |
>

## 2. 方案演进与瓶颈分析（共 4 次演进）

### 2.1 v1 同步竞价 baseline（原报告内容精简）

**链路特征**：客户端发起出价后，服务端在主链路内完成输入校验、Redis Lua 原子裁决、实时广播准备、排行榜更新、事件流写入、数据库落库，然后再向客户端返回 ACK。也就是说，用户等待的 ACK 同时承担“请求已处理”和“裁决已完成”的语义。

**关键瓶颈**：

- **Lua 临界区过重**：单次裁决需要在 Redis 单线程内连续执行多项读写、广播准备和榜单维护操作。Redis 单线程天然串行，临界区越大，排队越明显。
- **出价路径同步落库**：成交出价需要同步写入数据库。在高 QPS 下，数据库 IO 会直接拖慢用户收到 ACK 的时间。
- **排行榜同步刷新**：排行榜维护与主裁决抢占同一个 Redis 执行通道，使裁决链路更容易排队。

> 该阶段没有独立沉淀 v1 压测样本；后续优化均基于 v2 重测结果判断收益。具体数字在 2.2 节给出。

### 2.2 v2 同步竞价优化：异步落库 + 排行榜异步 + Lua 精简（原报告 T1–T5 沉淀）

**核心改造**：

1. 主路径减少额外状态读取，将裁决所需字段合并到一次 Redis Lua 执行中；
2. 广播准备、活跃流维护、排行榜更新和事件流写入从 Lua 原子段移出，改由异步消费链路处理；
3. 数据库落库改为批量消费、批量写入和批量确认，避免每次出价同步等待数据库；
4. 服务侧增加实时状态缓存、安全的过期请求预拒绝，以及最高在途价格保护，把明显无效的请求挡在 Redis 裁决之前；
5. Redis 按实时裁决、异步 worker、广播、排行和缓存等职责做隔离，降低不同流量之间的互相干扰。

**v2 阶段 5 次压测核心数字**（保留原报告关键表，用于与 v3/v4 横向比较）：

| 档位 | 入口 QPS | accepted/s | ACK P50 | ACK P95 | ACK P99 | Redis Lua P95 (ms) | 备注 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| T1 单拍品 500 QPS | 497.9 | 170.1 | 0.7ms | 6.1ms | 11.7ms | 7.5 | 单分片单线程线性区 |
| T2 单拍品 1000 QPS | 992.9 | 103.4 | 0.6ms | 10.1ms | 26.5ms | 18.6 | 排队开始抬头 |
| T3 单拍品 1500 QPS | 1493.7 | 71.1 | 0.6ms | 15.0ms | 49.8ms | 23.4 | 单分片单线程接近上限 |
| T4 5 拍品 × 500（2500 总） | 2493.1 | 188.3 | 0.7ms | 38.3ms | 99.2ms | 49.2 / 54.6 | 横向扩散有效推开天花板 |
| T5 30 拍品 × 100（压垮） | 3786（峰 9624） | 1328.3 | 6.8ms | 1068.4ms | 1106.5ms | 53.5 / 54.8（Redis Lua 未压垮） | 瓶颈转移到缓存实例慢操作 |

**v2 残留瓶颈**：

- **单拍品高压下 ACK 延迟仍随并发恶化**：T2/T3 单拍品场景下，ACK P95 从 6.1ms（T1 500 QPS）抬到 15.0ms（T3 1500 QPS），Redis Lua P95 从 7.5ms 抬到 23.4ms，表现出典型的客户端排队曲线。即便服务侧能预拒绝无效请求，**单 Redis 分片单线程仍是单拍品上限**，无法仅靠增加业务进程解决。
- **ACK 与裁决耦合**：客户端必须等 Redis Lua 真正裁决完成后才能收到 ACK。任何 Redis 抖动都会直接污染用户感知。
- **缓存慢操作反向拖慢主链路**：T5 中，30 个拍品 × 100 用户的黑名单清理流量使缓存实例出现慢操作，相关阶段 P95 达 794ms，并进一步把主链路 ACK P95 拉高到 1068ms。

### 2.3 v3 异步竞价 — 单拍品单 Kafka 分区，单 worker 顺序消费

**设计动机**：

- 将“用户感知的 ACK”和“系统裁决结果”彻底解耦：出价请求先写入 Kafka，客户端立即收到“入队成功”的 ACK；裁决由独立 worker pool 异步消费消息后调用 Redis Lua 完成。
- 继续保证顺序性：同一个拍品的出价必须串行裁决，才能保证当前价格单调递增。v3 采用最直接的方案：一个拍品路由到一个固定 Kafka 分区，一个分区由一个 worker 串行消费。

**实测瓶颈（Stage A：2 个拍品 × 500 QPS = 1000 总 QPS）**：

| 指标 | 实测值 | 解读 |
| --- | --- | --- |
| ACK QPS | 1000.04（几乎全部入队） | ACK 路径未被压垮，Kafka 写入正常。 |
| **ACK P95 / P99** | **22.4ms / 24.7ms** | ACK 只测“入队”耗时，主要由 Kafka 写入耗时决定。 |
| **Result P95 / P99** | **3.04s / 4.87s** | 端到端裁决延迟严重恶化，用户需要等待数秒才知道“成交 / 失败”。 |
| Decision 耗时 P95 | 4.2ms | worker 单次裁决本身很快。 |
| 成交裁决速率 | **6.96/s** | 实际成交速率。 |
| 拒绝裁决速率 | 475.5/s | 业务规则拒绝速率。 |
| **总裁决吞吐** | **约 482/s** | 成交与拒绝之和，**远低于 1000/s 入口流量**。 |
| **Queue reject** | **496.5/s** | 入队失败或内存队列满导致 ACK 阶段直接拒绝；超过吞吐能力的那部分请求在这里被背压挡住。 |
| 裁决排队 | 平均 785，峰值 968 | 内存队列长期接近满载（容量 1000）。 |
| CPU | 平均 63%（峰 127%） | 单 worker 串行，CPU 远未压满。 |
| Goroutines / RSS | 3096 / 417 MB | 协程数与内存水位健康。 |

> 数据来源：Stage A 实测窗口 `2026-06-09T05:51:58.843Z` ~ `2026-06-09T05:57:01.282Z`，按 10s 粒度聚合。

**瓶颈定位**：

- **单分区、单 worker 串行成为新瓶颈**：v3 把 Kafka 分区数与拍品数量绑定，每个分区由一个 worker 串行消费。单 worker 的裁决吞吐天花板约 480/s 量级，与 1000/s 入口流量相差约 2 倍。
- **超额请求在 ACK 阶段被背压拒绝**：Queue reject 速率约等于入口流量与实际裁决吞吐之差（497 ≈ 1000 − 482），符合“内存队列满后直接拒绝”的行为。
- **Result 延迟主要来自排队等待**：单次裁决 P95 只有 4.2ms，但 Result P95 达到 3.04s，说明时间并未消耗在裁决执行本身，而是消耗在 Kafka 与服务内队列排队。
- **CPU 没压满不是好消息**：v3 不是 CPU 瓶颈，而是架构强制串行把吞吐天花板钉死。单纯增加 CPU 或业务实例，无法解决同拍品绑定单 worker 的串行限制。

**结论**：v3 成功把 ACK 路径和裁决路径解耦，ACK 延迟从 v2 单拍品高 QPS 下的几十毫秒压回到约 22ms 的稳定平台；但裁决吞吐天花板从“单 Redis 分片单线程”转移成“单 Kafka 分区单 worker”，并没有真正打开横向扩展空间。因此需要 v4 进一步解耦“分区数”和“顺序性”。

### 2.4 v4 异步竞价 — 16 分区随机分发 + Redis 同分片 Lua 保证顺序性

**设计动机**：

v3 的问题不在于异步裁决方向错误，而在于把“顺序性”硬绑定到了“Kafka 分区 / worker”上。v4 的核心调整是将两者拆开：

- **Kafka 分区只服务吞吐**：消息随机分发到 16 个 Kafka 分区，不再按拍品固定落到某个分区，让所有 worker 可以并行消费。
- **同拍品顺序性由 Redis 兜底**：同一个拍品的裁决键落在同一个 Redis 分片，Redis Lua 仍按单线程串行执行。也就是说，worker 可以并发拉取同拍品消息，但真正修改价格状态时仍由 Redis 按到达顺序排队裁决。
- **工作池批量确认**：worker pool 默认容量为 32，并采用批量提交方式降低 Kafka 确认开销。

**Stage B 实测（2 个拍品 × 500 QPS = 1000 总 QPS）**：

| 指标 | 实测值 |
| --- | --- |
| ACK QPS | 1000.01 |
| **ACK P50 / P95 / P99** | **9.0ms / 25.1ms / 46.2ms** |
| **Result P50 / P95 / P99** | **37.1ms / 106.7ms / 156.9ms** |
| Decision P95 | 13.7ms |
| 裁决排队 | 平均 42，峰值 107 |
| **Kafka lag** | **0**（全程零积压） |
| Worker inflight | 平均 27.8 / 32（接近满负荷但未阻塞） |
| **Queue reject** | **0**（无背压） |

| CPU | 平均 271%（约 2.7 核） |
| Goroutines / RSS | 3123 / 447 MB |

#### v4 与 v3 横向对比

| 维度 | v3（Stage A，1000 QPS） | v4（Stage B，1000 QPS） | v4（Stage C，2000 QPS） |
| --- | --- | --- | --- |
| ACK P95 | 22.4ms | **25.1ms** | **23.0ms** |
| ACK P99 | 24.7ms | 46.2ms | 26.2ms |
| Result P95 | **3040ms** | **107ms**（下降约 28 倍） | 1069ms |
| Result P99 | 4866ms | 157ms | 1536ms |
| Decision P95 | 4.2ms | 13.7ms | 10.4ms |
| Decision QPS | 482/s | **858/s**（提升约 1.8 倍） | **1314/s**（提升约 2.7 倍） |
| 裁决排队 | 785/1000（饱和） | 42/1000 | 984/1000（饱和） |
| Kafka lag | —（v3 未单独采集该口径） | 0 | 227（峰 413） |
| Queue reject | 497/s | 0 | 630/s |
| Worker inflight | 1（单 worker） | 27.8/32 | 32/32（打满） |
| CPU | 63% | 271% | 161% |
| Goroutines | 3096 | 3123 | 6131 |

**关键观察**：

- **吞吐拐点被 v4 推开约 2.7 倍**：v3 在 1000 QPS 入口下裁决吞吐 482/s，约一半请求被背压拒绝；v4 在同样入口下裁决 858/s，无背压、无积压；直到 2000 QPS 才再次饱和到 1314/s。
- **ACK 延迟基本不受裁决侧饱和影响**：v4 ACK P95 在 Stage B / C 分别为 25ms / 23ms，与 v3 的 22ms 同档。这说明 ACK 路径主要受 Kafka 写入耗时影响，不会因为后端 worker 是否打满而明显恶化。
- **Result 延迟直接反映裁决吞吐能力**：v3 Result P95 = 3040ms，是裁决跟不上、队列排队数秒的结果；v4 Stage B Result P95 = 107ms，说明 worker 池能消费掉入口流量；Stage C 退化到 1069ms，则说明测试已越过当前吞吐拐点。
- **Stage C CPU 低于 Stage B 并不代表还有余量**：Stage C 的服务内存队列已经饱和，超额请求在入队阶段被拒绝，不再进入 Kafka，也不消耗 worker CPU。同时 worker 长期 32/32 打满，更多时间等待 Redis Lua 往返，CPU 忙碌比例反而下降。这是背压保护后的资源水位，不是扩容空间。
- **单次裁决耗时保持稳定**：Decision P95 全程在 10–14ms，说明 Redis Lua 本身没有出现数量级恶化；用户侧 Result 变慢主要来自排队，而不是裁决执行变慢。

## 3. 当前架构性能总结

### 3.1 v4 当前稳定区间

- **稳定 ACK QPS（无背压）**：约 1000 QPS / 实例。Stage B 全程无 Queue reject、无 Kafka lag，ACK P95 为 25ms。
- **吞吐拐点**：约 2000 QPS / 实例。Stage C 已出现 worker pool 32/32 打满、裁决排队 984/1000、Kafka lag 平均 227 / 峰值 413、Queue reject 630/s。
- **裁决吞吐天花板**：当前架构约 1300–1400/s。Stage C 中成交 23/s、拒绝 1292/s，总裁决吞吐约 1314/s，瓶颈主要在 worker pool 规模与 Redis 同分片 Lua 串行带宽。
- **CPU 不是当前主瓶颈**：8 核机器在 Stage B 使用约 2.7 核，Stage C 受背压保护反而回落到约 1.6 核。下一步更有效的方向不是单纯加 CPU，而是扩展 worker pool，并评估 Redis 分片拆分。

| --- | --- | --- | --- | --- |

### 3.3 一致性不变量保留

v2 沉淀的一致性不变量在 v3/v4 异步链路下继续生效：

- **同拍品顺序性**：v2 由 Redis 单线程保证；v3 由 Kafka 分区顺序保证；v4 由 Redis 同分片 Lua 单线程兜底。v4 中 Kafka 消费可以并发，但最终修改同拍品价格状态时仍串行。
- **价格单调与原子裁决**：出价价格单调递增、Redis Lua 原子裁决、请求幂等保护、成交事件作为后续异步处理依据，以及排行榜乱序保护策略均保持不变。
- **ACK 超时与提交延迟**：Stage A/B/C 三段窗口期间未观察到 ACK 超时；Stage B/C 的提交延迟 P95 约 0.24s，符合批量提交的设计预期。

## 4. 后续优化方向（可选简短）

1. **扩展 worker pool，并评估 Redis Lua 拆分**：Stage C 已经被 32 个 worker 与 Redis 单分片 Lua 串行带宽卡住。下一步可评估将 worker pool 扩到 64 / 128，并按拍品维度拆分 Redis 分片，打开分片维度的并发空间。
2. **让裁决排队容量更自适应**：当前队列容量为固定 1000。Stage C 饱和后很快出现 Queue reject。后续可根据 worker pool 在途任务数动态调整队列容量，或在溢出时引导客户端退避重试。
3. **继续关注 Result 推送链路**：Stage B/C 的结果推送耗时 P95 稳定在约 0.95ms，说明推送通道本身不是瓶颈。Stage C 用户感知 Result 延迟超过 1s，根因仍是裁决排队，不是推送慢。
4. **客户端引入 backoff（退避）策略**：v2 T5 已验证客户端无退避会放大雪崩风险。生产场景需要客户端 SDK 引入指数退避和 jitter（随机抖动），避免 ACK 变慢时集中重试。
5. **保留同步链路的剩余优化项**：如果生产中仍存在 v2 同步链路或混合模式，需要继续处理缓存实例慢操作问题，例如对黑名单状态做短 TTL 缓存、LRU 控制和增量失效，避免缓存慢操作反向拖慢主链路。
