# 出价链路压测优化复盘

> 这份复盘不再按每一轮压测截图流水记录，而是按关键优化过程整理。重点回答两个问题：
>
> 1. 为了把出价 QPS 拉高，我到底改了哪些地方。
> 2. 在把同步链路不断变短、把功能不断异步化的过程中，如何保证拍卖一致性和用户实时体验不被破坏。

## 1. 一句话结论

这次优化的核心不是“让所有请求都成交”，而是把实时拍卖链路拆清楚：

```text
强一致裁决：Redis Lua，只负责当前价、领先者、状态、版本、序号、幂等
低成本拒绝：Go 本地缓存 + 安全预拒绝 + In-Flight Gate
实时展示：ACK / PubSub / WebSocket / Redis state
最终记录：Redis Stream / Kafka / MySQL worker
可延迟展示：排行榜、在线人数、历史出价
```

最终方向是：同步 ACK 只等待必须强一致的裁决；其他广播、排行榜、落库、在线人数、Stream 裁剪全部从主原子段和主 Redis 热路径里移出去。

## 2. 压测目标的重新定义

最开始我容易把“发送 QPS”和“成交 QPS”放在一起看，后来确认这两个指标不能简单等同。

单拍品拍卖天然有串行推进特征：

```text
当前价 1000
1000 个用户同时出 1100
只有第一个能成交
后面 999 个即使系统处理很快，也会因为当前价已变成 1100 而被拒绝
```

所以压测要分成三种模型：

| 模型 | 目的 | 关键指标 |
|---|---|---|
| 拒绝压力模型 | 大量 stale / 低价请求，验证 Go 预拒绝、ACK、Redis 排队 | rejected QPS、ACK P95、Lua enter 比例 |
| 成功推进模型 | 尽量基于最新价格出下一口价，验证成交链路 | accepted QPS、广播延迟、Stream/Kafka/MySQL lag |
| 混合模型 | 模拟真实直播间：少量成功推进 + 大量跟价失败 | accepted QPS、reject reason 分布、用户侧实时性 |

这次优化后，我更关注：

- `lua_enter_total / bid_request_total`：多少请求真正进入 Lua。
- `accepted_total / lua_enter_total`：进入 Lua 的请求里有多少是有效竞争。
- ACK P95/P99：用户感知延迟。
- Redis `rt-*` EVALSHA P95：主裁决 Redis 是否还在排队。
- `rt-worker-* / pubsub-* / ranking-* / cache` 是否从主链路隔离出去。

## 3. 第一阶段：先让压测模型可信

最早遇到的问题不是后端性能，而是压测流程和业务接口不一致。

我先把压测工具调整成真实买家路径：

```text
登录
  -> 获取直播场次
  -> 获取当前拍品
  -> 报名 / 保证金准备
  -> 建立 WebSocket
  -> 等待 snapshot / 当前价
  -> 按 incrementRule 计算出价
  -> 发送 bid.place
```

关键修正：

- 出价消息必须按后端协议携带 `auctionId`、`price`、`expectedCurrentPrice`。
- 压测工具不能自造“加价分”，必须读取后端返回的 `incrementRule`。
- `expectedCurrentPrice` 必须来自 ACK、广播或 snapshot，否则大量 `BELOW_MIN_INCREMENT` 是压测模型造成的。
- 不能只看 goTest 的发送 QPS，还要和后端 Prometheus 接收 QPS 对齐，判断是客户端没发满还是后端吃不下。

这一阶段的意义是：后续看到拒绝、延迟、成交下降时，先排除“压测工具自己打错”的干扰。

## 4. 第二阶段：观测先行

在 1000 QPS 以后，只看接口总耗时已经不够。我补了两类观测。

### 4.1 Go 业务阶段打点

核心指标：

```text
aieas_auction_bid_stage_duration_seconds{stage,result}
aieas_auction_bid_route_total{route,result}
```

我把出价链路拆成这些阶段：

| stage | 含义 |
|---|---|
| `input_validate` | 参数校验 |
| `auction_snapshot_source` | 拍品快照来自 cache 还是 db |
| `bid_prerequisites` | 报名/保证金前置校验 |
| `pre_reject_local` | Go 本地预拒绝 |
| `bidder_nickname` | 昵称补全 |
| `realtime_place_bid` | Redis Lua 原子裁决 |
| `enrich_result` | ACK 结果补字段 |
| `direct_publish_result` | 非 Stream 模式下直接广播 |
| `sync_persist_bid` | 非 Stream 模式下同步落库 |

有了阶段打点后，我能判断：

- ACK 高是不是卡在 Lua。
- 是否还有 DB 查询漏进热路径。
- Go 预拒绝到底拦了多少。
- `bid_prerequisites`、昵称、Agent hook 是否在同步路径里拖慢。

### 4.2 Redis 客户端与服务端双视角

Redis 需要同时看两类指标：

```text
客户端视角：redis_command_duration_seconds / redis_pool_wait
服务端视角：INFO commandstats / SLOWLOG / INFO cpu / Stream lag
```

压测中出现过一个典型现象：

- Redis server `usec_per_call` 不高。
- 但 Go 侧 EVALSHA P95 很高。

这说明瓶颈不一定是单条 Lua 真执行了几十毫秒，也可能是：

- Redis 单线程队列排队。
- 同一 Redis server 上 PubSub / worker / ranking / online counter 抢执行时间。
- 客户端连接池等待。
- Stream、ZSet、Hash 数据规模增长导致尾部延迟变差。

因此后续优化不能只盯 Lua 代码，也要拆 Redis pool、Redis server 责任和后台 worker 行为。

## 5. 第三阶段：拒绝路径轻量化

高 QPS 单拍品下，大部分请求天然会失败。失败本身不是问题，问题是失败路径不能太重。

优化前，拒绝请求曾经也会走：

```text
拒绝结果
  -> 广播
  -> Stream
  -> Kafka
  -> MySQL bid_record
```

这会让大量无效请求拖慢整个系统。

优化后：

```text
拒绝请求
  -> Go / Lua 得到拒绝原因
  -> ACK 给本人
  -> 指标计数
  -> 结束
```

只保留成功出价进入可靠事件链路：

```text
accepted
  -> Redis state 更新
  -> Redis Stream 事件
  -> PubSub / WebSocket
  -> Kafka / MySQL worker
  -> ranking worker
```

一致性影响：

- 拒绝请求不改变拍卖状态，所以不需要进入 Stream/MySQL 才能保证状态一致。
- 拒绝原因通过指标保留，满足压测和运维观测。
- 成功请求仍然有可靠事件链路，满足审计和历史记录。

## 6. 第四阶段：MySQL 退出实时裁决路径

压测中出现过这些慢查询：

```sql
SELECT * FROM auction_lot WHERE auction_id = ? LIMIT 1;
SELECT * FROM bid_record WHERE auction_id = ? ORDER BY bid_price DESC LIMIT 1;
```

这类查询不应该出现在实时出价主路径里。

优化后职责划分：

| 数据 | 实时路径来源 |
|---|---|
| 当前价 | Redis state |
| 领先者 | Redis state |
| 拍卖状态 | Redis state |
| 起拍价 / 封顶价 | Redis state + Go 短 TTL 快照 |
| 加价规则 | Redis state + Go 短 TTL 快照 |
| 成交历史 | Redis Stream / Kafka / MySQL 异步 |
| 历史最高价查询 | MySQL / ranking cache，不参与裁决 |

一致性影响：

- 当前价和领先者不再从 MySQL 推导，避免异步落库延迟反向影响实时裁决。
- MySQL 仍然是历史事实和审计事实，但不是“当前拍卖状态”的同步来源。
- 落锤用 Redis state 的 `leader_bidder_id/current_price`，不依赖异步排行榜。

## 7. 第五阶段：Go 预拒绝与 In-Flight Gate

Lua 是强一致裁决点，但并不代表所有请求都应该进入 Lua。

我补了三类 Go 侧预拒绝。

### 7.1 本地实时状态缓存

本地缓存从只保存：

```text
CurrentPrice / Status / Version / EndTime
```

扩展到能支持完整价格预判：

```text
CurrentPrice
StartPrice
CapPrice
IncrementAmount
MaxBidSteps
Status
Version
EndTsMs
```

这样 Go 可以直接拒绝明显不可能成功的请求：

- 低于起拍价。
- 超过封顶价。
- 不符合加价步长。
- 超过最大加价步数。
- 低于当前最低有效价。

一致性边界：

- Go 预拒绝只能拒绝“无论 Redis 当前价是否更新都不可能成功”的请求。
- Go 不能直接接受请求，所有 accepted 必须经过 Lua。
- 本地缓存来自 Lua 返回结果和 Redis state，状态只用于剪枝，不作为最终裁决。

### 7.2 Safe Stale Reject

这个规则只拒绝必败请求：

```text
如果客户端 expectedCurrentPrice 已低于本地 currentPrice
并且 price 还低于 currentPrice + step
那么这笔请求在 Redis 中一定也无法成功
```

原因是拍卖当前价只会单调上升，不会回退。

一致性边界：

- 这类拒绝不会误杀本来能成功的追价。
- 它只是把 Lua 里必然 `BELOW_MIN_INCREMENT` 的请求提前挡掉。

### 7.3 Highest In-Flight Gate

单拍品 1000 QPS 时，最大浪费来自大量用户同时出同一个价。

最初的 Gate 是同价位最多放行 2 个进入 Lua。后来进一步改成：

```text
每个拍卖记录当前正在进入 Lua 的最高价

低于这个最高价的请求：
  -> Go 直接 AUCTION_BUSY

高于或等于这个最高价的请求：
  -> 每个价位最多放行 2 个进入 Lua
```

这个策略的目标不是提高“接受率”，而是减少无效 Lua 排队。

一致性边界：

- Gate 只会拒绝为 `AUCTION_BUSY`，不会制造 accepted。
- 被 Gate 拒绝的客户端需要基于最新 ACK/广播/snapshot 重试。
- 每个价位仍放行 2 个，避免第一个请求因为异常、幂等、频控导致有效出价完全被挡住。
- Lua 返回后释放 slot，防止长期占用。

这个优化能显著降低 Lua enter 比例，但如果压测模型仍然基于旧价格发大量请求，accepted QPS 不会线性升高。它解决的是“无效请求挤爆 Lua”，不是改变单拍品成交串行推进的业务事实。

## 8. 第六阶段：资格、昵称、快照缓存

### 8.1 报名/保证金前置校验

后来我把报名和保证金从 Lua 中移出，改为 Go 层前置校验，并加本地 30s TTL 缓存。

适用前提：

- 进入出价链路的用户基本已报名。
- 保证金状态在拍卖运行中很少变化，或业务上不可逆。
- 如果有资格撤销，必须主动失效缓存，或把校验重新放回 Lua。

一致性边界：

- 资格校验是业务门面，不负责更新当前价。
- 通过 Go 校验后，Lua 不再重复查资格集合。
- 这个设计牺牲了资格变更的即时强一致，换取主 Lua 更短。
- TTL 和风控开关用于压测与线上策略控制。

### 8.2 昵称缓存

昵称只是展示字段，不应该每次出价同步查用户表。

处理方式：

- 出价时尽量从本地缓存取昵称。
- 缓存 miss 再查用户仓储。
- 成交状态和当前价不依赖昵称。

实时性影响：

- 昵称短暂旧值不影响拍卖裁决。
- ACK 和广播仍然能返回核心出价结果。

### 8.3 拍品快照缓存

拍品静态字段如起拍价、封顶价、加价规则，用短 TTL 缓存。

一致性边界：

- 当前价、领先者、版本不从这个缓存裁决。
- 拍卖运行中如果允许修改规则，需要同步刷新 Redis state 和本地缓存。
- 默认规则在运行中稳定，因此缓存是可接受的。

## 9. 第七阶段：Lua 主原子段瘦身

Redis Lua 只保留必须原子完成的事情。

### 9.1 保留在 Lua 内的职责

当前 `bid.lua` 负责：

- 幂等 GET/SET。
- 读取拍卖 state。
- 校验状态、价格、步长、封顶价、最大加价步数。
- 频控计数。
- 更新当前价、领先者、结束时间、版本、bid_count。
- 生成全局递增 seq。
- 写入 accepted Stream 事件。
- 返回 ACK 所需的裁决结果。

这些必须在同一个原子段里完成，否则会出现并发用户同时成交、当前价回退、重复 requestId 产生多条记录等问题。

### 9.2 从 Lua 移出的职责

| 移出项 | 新位置 | 原因 |
|---|---|---|
| `PUBLISH` | Go 侧异步 PubSub | 广播是 best-effort，不应阻塞 ACK |
| 排行榜 `ZADD/HSET` | ranking worker | 排行榜允许短暂延迟 |
| 报名/保证金 `SISMEMBER` | Go + TTL cache | 减少主 Lua Redis call |
| `TYPE` 调试检查 | 测试 / 巡检 | 热路径不做调试检查 |
| `XADD MAXLEN` | Stream trim worker | 裁剪不需要跟裁决同原子 |
| 完整 JSON 返回 | RESP 数组返回 | 减少 Lua `cjson.encode` 和 Go JSON decode 成本 |

### 9.3 固定加价规则快路径

固定加价规则提前解析并写入 Redis state：

```text
increment_rule_type = fixed
increment_fixed_amount = amount
```

Lua 看到固定规则时直接走数值字段，不再 `cjson.decode(increment_rule)`。

一致性边界：

- Redis state 仍然是 Lua 的规则来源。
- 如果规则是 ladder，Lua 仍能回退到解析 JSON。
- 初始化拍卖时写入的规则字段必须和拍品配置一致。

### 9.4 返回数组而不是完整 JSON

Lua 返回由固定下标组成的数组：

```text
requestId, auctionId, liveSessionId, bidderId, price, accepted, reason, currentPrice, ...
```

Go 侧按下标解析成 `domain.BidResult`。

一致性和兼容性：

- 对外 HTTP/WebSocket ACK 仍然是 Go marshal 后的原 JSON 结构。
- PubSub 仍发布 JSON object，前端协议不变。
- Go 侧保留旧 JSON 对象解析能力，兼容历史幂等缓存。
- Redis 幂等缓存改存 compact 数组 JSON，重复请求会还原为数组并补 `duplicate=true`。

## 10. 第八阶段：异步链路拆分

### 10.1 成功事件是异步链路的源头

accepted 出价在 Lua 内写入 Redis Stream，这是异步链路的源头：

```text
Lua accepted
  -> XADD bid stream
  -> ACK 返回给用户
  -> PubSub / WebSocket 广播
  -> Kafka bridge
  -> MySQL writer
  -> ranking worker
```

一致性边界：

- Stream 事件和 Redis state 更新在同一个 Lua 原子段里。
- 只要 Lua accepted 成功，就一定有对应 Stream 事件。
- MySQL、排行榜、Kafka 可以延迟，但不能反向影响 ACK。

### 10.2 Kafka bridge 批量化

Kafka bridge 从逐条慢节奏改成：

- 批量 XREADGROUP。
- 批量 `WriteMessages`。
- interval 从 1s 降到 50ms。
- count 从 64 提到 256。
- Kafka 成功后批量 ACK Redis Stream。

一致性边界：

- Kafka 写失败不 ACK，事件留在 PEL 中重试。
- 超过重试阈值进入 DLQ。
- MySQL writer 使用幂等写入，避免重复消费产生重复记录。

### 10.3 排行榜异步化

排行榜从 Lua accepted 主路径移出，由独立 consumer group 更新。

为防止乱序：

```text
按 price、bidTsMs、seq 比较
只有新事件更“靠后”才覆盖旧排名
```

一致性边界：

- 当前最高价不依赖排行榜，而是依赖 Redis state。
- 落锤不读排行榜，而是读 state leader/currentPrice。
- 排行榜只是展示视图，允许短暂延迟。
- TopN 读取 ranking Redis，不影响主出价 Redis。

### 10.4 Stream 裁剪异步化

`XADD MAXLEN ~ 10000` 从 Lua 移出，改成后台 worker 每秒执行：

```text
XTRIM auction:<id>:stream MAXLEN ~ 10000
```

一致性边界：

- 裁剪只影响历史 Stream 保留长度，不影响当前价和领先者。
- 裁剪长度必须大于 worker lag 和补偿回放窗口。
- 需要持续监控 `XPENDING`、consumer lag、DLQ。
- 如果 lag 变大，应先扩 worker 或提高保留长度，不能盲目裁剪。

## 11. 第九阶段：Redis pool 与 Redis server 隔离

单独拆连接池只能解决客户端连接等待，不能解决同一 Redis server 单线程执行时间竞争。

所以我把 Redis 职责拆成：

| 指标标签 | 职责 |
|---|---|
| `rt-*` | 主出价 Lua、拍卖 state、Stream 写入 |
| `rt-worker-*` | Stream XREADGROUP、XACK、XTRIM、DLQ |
| `pubsub-*` | PubSub 发布和订阅，落到 Redis-Cache 实例 |
| `ranking-*` | 排行榜 ZSet/Hash，落到 Redis-Cache 实例 |
| `cache` | HTTP 幂等 store、在线人数、配置/黑名单等缓存；出价 requestId 幂等仍在 `rt-*` Lua 内 |

关键变化：

- PUBLISH 不再打 `rt-*`。
- ranking EVALSHA 不再打 `rt-worker-*` 对应的 RT server。
- 在线人数从 RT 移到 Cache。
- WebSocket Touch 增加 30s 本地节流，避免每次心跳都打 Redis。

一致性边界：

- `rt-*` 仍然是强一致裁决源。
- `pubsub-*` 丢消息不影响状态，客户端可通过 snapshot/replay 收敛。
- `ranking-*` 延迟不影响当前最高价。
- `cache` 在线人数允许延迟展示。

## 12. 实时性是怎么保证的

提高 QPS 时，我把“实时性”拆成三层。

### 12.1 ACK 实时性

用户出价是否成功，以 Redis Lua 返回为准。

```text
客户端 bid.place
  -> Go 预校验
  -> Redis Lua
  -> ACK
```

ACK 不等待：

- MySQL 落库。
- Kafka。
- 排行榜。
- PubSub 成功。
- 在线人数。
- Stream 裁剪。

这保证了用户最核心的反馈路径最短。

### 12.2 当前价实时性

当前价有三个收敛来源：

1. 出价者收到 ACK。
2. 房间用户收到 PubSub/WebSocket 广播。
3. 客户端重新拉 snapshot/state。

PubSub 是低延迟 best-effort；Redis state 是最终实时状态。

如果 PubSub 丢了：

- Stream relay / replay 可以补事件。
- 客户端重新进房或刷新 state 可以拿到最新当前价。
- 当前价不会因为广播丢失而回退。

### 12.3 展示视图实时性

排行榜、在线人数、历史出价属于展示视图。

处理原则：

- 允许短暂延迟。
- 不作为裁决依据。
- 用 worker、Cache、定时刷新、WebSocket 广播异步更新。

这样用户看到的“当前最高价”仍然实时，而“排行榜第几名、在线多少人、历史记录是否已经落库”可以稍后收敛。

## 13. 一致性是怎么保证的

### 13.1 当前价单调性

Redis Lua 在一个原子段内完成：

```text
读取 current_price
校验 price
更新 current_price / leader_bidder_id / version
生成 seq
写 Stream
返回 result
```

Redis 单线程执行 Lua，保证同一个拍卖不会同时接受两笔冲突出价。

### 13.2 幂等一致性

同一个 `requestId` 的处理在 Lua 内完成：

```text
GET idem_key
如果存在，直接返回原结果并标记 duplicate
如果不存在，正常裁决后 SET idem_key
```

幂等和裁决在同一个 Lua 里，避免 Go 先查、Lua 后写之间的竞态。

TTL 默认 30s，覆盖 WebSocket 短时间重试；长期审计依赖 accepted Stream 和 MySQL bid_record。

### 13.3 accepted 事件一致性

accepted 时，Redis state 更新和 Stream 写入同一个 Lua 原子段完成。

这意味着：

- 有 accepted ACK，就应该有 accepted Stream。
- Worker 可以晚消费，但事件不会因为 Go 进程崩溃而丢。
- MySQL 最终记录可以通过 Stream/Kafka/reconciler 补偿。

### 13.4 预拒绝不会产生错误成交

Go 预拒绝只做剪枝：

- 它可以拒绝必败请求。
- 它不能接受请求。
- 所有 accepted 必须来自 Lua。

如果 Go 本地状态偏旧，最多少拒绝一些请求，让它们进入 Lua；不会把本该失败的请求直接判成功。

### 13.5 资格校验的一致性取舍

报名/保证金移到 Go 后，一致性前提变成：

- 报名/保证金状态在拍卖运行期稳定。
- 状态写入时同步更新 Redis / cache。
- 状态变更需要主动失效本地缓存。

如果业务以后要求“拍卖进行中可实时撤销资格”，这块必须重新收紧：要么缩短 TTL 并主动失效，要么把关键资格校验放回 Lua。

### 13.6 排行榜不参与裁决

排行榜异步化后，必须坚持：

```text
当前价 / 领先者 = Redis state
排行榜 = 展示视图
```

因此：

- Hammer 不读 ZSet。
- 最高价展示优先用 state / ACK / broadcast。
- TopN 可以短暂延迟。

这就是排行榜能从主 Lua 移出去的前提。

## 14. 为什么 Go 已经拦住很多请求，成交 QPS 仍不一定升高

这个问题在 1000 QPS 压测里很明显。

Go 拦住 80%-90% 请求，说明无效请求不再大量进入 Lua，这是好事。但成交 QPS 是否升高，还取决于压测模型是否能持续产生“基于最新当前价的下一口价”。

如果客户端仍然大量基于旧状态发：

```text
当前价已经 1500
客户端还在发 1300 / 1400 / 1500
```

那么 Go 会把它们拒绝成：

```text
AUCTION_BUSY
BELOW_MIN_INCREMENT
STALE_AUCTION_STATE
```

这会降低 Lua 压力，但不会凭空增加可成交请求。

想让“1000 QPS 输入下成交 QPS 也好看”，需要压测工具配合：

- 更快消费 ACK/广播。
- 用最新 currentPrice 生成下一口价。
- 控制并发窗口，避免大量旧状态请求堆积。
- 或者用多拍品分摊竞争，而不是所有流量打一个拍品。

单拍品成交推进本质上仍然受业务规则、价格步长、客户端反馈延迟影响。

## 18. 最终复盘

这次优化的核心经验是：实时拍卖不能只靠“把 Redis QPS 打高”来证明性能。

真正要做的是把链路拆开：

```text
能本地安全拒绝的，不进 Lua
必须强一致的，留在 Lua
可延迟展示的，出 Lua 后异步
可最终一致的，走 Stream/Kafka/MySQL
可丢可补的，走 PubSub + replay/snapshot
```

提高 QPS 的过程，本质上是在不断缩短同步 ACK 路径。但每移走一块逻辑，都必须回答它的一致性来源是什么：

- 当前价的一致性来自 Redis state + Lua 原子更新。
- accepted 事件的一致性来自 Lua 内 XADD。
- 幂等一致性来自 Lua 内 GET/SET。
- 排行榜一致性来自 accepted Stream + seq 防乱序。
- MySQL 一致性来自 Stream/Kafka worker 幂等写和补偿。
- PubSub 实时性来自异步广播，可靠性由 Stream replay/snapshot 补齐。

只要这些边界清楚，系统就可以在提升 QPS 的同时保持拍卖状态正确，并把用户真正敏感的 ACK 和当前价展示保持在低延迟路径上。
