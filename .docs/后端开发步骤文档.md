# 后端开发步骤文档

> 本文基于 `docs/需求文档-后端.md`、`docs/后端详细技术设计.md`、`docs/数据库设计.md`、`docs/API文档.md` 与 `docs/ddl.sql` 整理，用于指导「实时竞拍大师」后端按 MVP 范围落地开发。当前目标是 3 周内完成可演示的竞拍闭环：商品管理、拍品管理、报名保证金、WebSocket 实时出价、动态排名、反狙击延时、唯一落锤、订单生成、风控与审计。

## 1. 开发前准备

1. 明确技术基线：Go 后端以 CloudWeGo Hertz 承载 HTTP REST 与 Hertz WebSocket，Redis 负责竞拍热路径，MySQL 负责最终事实表。
2. 对齐业务约束：
   - 单房间在线人数目标：`1k ~ 5k`。
   - 单房间出价峰值：`100 ~ 500 QPS`。
   - 出价链路 P99：`< 300ms`。
   - 落锤成交必须 `100%` 唯一，禁止重复成交、重复扣保证金、重复生成订单。
   - 所有金额字段使用整数「分」，数据库统一使用 `BIGINT`。
   - 起拍价允许为 `0`，但不可小于 `0`。
3. 统一协议约定：
   - REST 路径版本统一使用 `/api/v1`。
   - 通用响应结构为 `{code,message,data,trace_id}`。
   - 除登录、公开列表等接口外均携带 `Authorization: Bearer <jwt_token>`。
   - REST 关键写操作携带 `Idempotency-Key`；WebSocket 出价使用 `requestId` 做幂等与响应关联。
   - WebSocket JSON 信封统一为 `{type,payload,requestId,seq}`。
4. 明确角色与权限：买家负责浏览、报名、出价、支付；商家负责商品与拍品管理、开拍、落锤；管理员负责审核、用户、黑名单、风控规则、审计和看板。
5. 开发前先阅读并确认以下资料：
   - `docs/API文档.md`：接口路径、请求响应、错误码、WS 事件。
   - `docs/后端详细技术设计.md`：架构、目录、Redis Lua、Hub、调度与部署。
   - `docs/数据库设计.md` 与 `docs/ddl.sql`：表结构、索引、状态机、Redis Key。
   - `docs/需求文档-后端.md`：功能清单、3 周排期、验收标准与风险。

## 2. 项目结构与分层

推荐按 Go 标准 layout 演进，业务代码收敛到 `internal`，基础可复用能力放到 `pkg`，脚本和迁移独立管理。

```text
backend/
├── cmd/server/                 # 服务入口，手工装配配置、DB、Redis、Hub、Service、Router
├── configs/                    # YAML 配置与环境覆盖
├── internal/
│   ├── app/                    # Hertz Server 生命周期与路由注册
│   ├── config/                 # 配置加载与校验
│   ├── middleware/             # JWT、RequestID、Recovery、幂等、限流、审计
│   ├── domain/                 # 领域模型、状态枚举、业务错误码
│   ├── repository/             # MySQL 访问层：user/item/auction/bid/order/deposit 等
│   ├── modules/                # 模块 application 层：auth/auction/bid/hammer/deposit/risk 等
│   ├── service/                # 兼容性测试包装层，非主业务实现
│   ├── transport/http/         # REST Handler 与 DTO 绑定校验
│   ├── transport/ws/           # WebSocket Handler、Hub、Room、Client、Dispatcher
│   ├── scheduler/              # 自动落锤、重启恢复、补偿任务
│   └── observability/          # logger、metrics、health
├── pkg/
│   ├── redis/                  # go-redis/v9 客户端、SCRIPT LOAD/EVALSHA/NOSCRIPT 回退
│   ├── jwt/                    # JWT 签发与校验
│   ├── idgen/                  # ID 生成
│   └── timeutil/               # 时钟抽象，便于测试
├── scripts/lua/                # bid.lua、hammer.lua
├── migrations/                 # MySQL 迁移 SQL
├── deployments/                # Dockerfile、docker-compose.yaml
├── go.mod
└── go.sum
```

分层职责建议如下：

| 层级 | 职责 | 注意事项 |
| --- | --- | --- |
| `transport/http` | 参数绑定、鉴权上下文读取、响应封装 | 不直接写 Redis/MySQL，不承载复杂业务判断 |
| `transport/ws` | WS 升级、连接管理、房间订阅、消息分发、广播 | 房间内通过 goroutine/channel 串行化广播，避免并发 map 写 |
| `service` | 业务编排、状态机迁移、事务边界、调用 Redis Lua | 出价和落锤的权威判断放 Redis Lua；MySQL 作事实落库 |
| `repository` | GORM/sqlc 数据访问 | 唯一键冲突按幂等语义处理，查询必须走设计索引 |
| `domain` | 枚举、错误码、领域对象 | 与 API 错误码和数据库状态保持一致 |
| `scheduler` | 到点落锤、重启恢复、补偿任务 | 所有任务带 `context.Context`，支持优雅退出 |
| `pkg` | 基础设施封装 | 不泄漏业务上下文，便于单测 |

## 3. 环境与依赖

1. Go：设计文档推荐 Go `1.22+`，当前仓库 `go.mod` 使用 Go `1.26.2`，开发环境需与 `go.mod` 对齐或在团队确认后统一调整。
2. HTTP/WS：
   - HTTP 框架：`github.com/cloudwego/hertz`。
   - WebSocket：`github.com/hertz-contrib/websocket`。
3. Redis：Redis 7 + `github.com/redis/go-redis/v9`，用于热状态、幂等、频控、排行榜、分布式落锤锁与事件流。
4. MySQL：MySQL 8，字符集 `utf8mb4`，时间字段使用 `DATETIME(3)`。
5. ORM/SQL：优先使用 GORM，或按团队决定使用 sqlc/sqlx；无论选型如何，唯一索引和事务语义必须与 DDL 保持一致。
6. 日志与指标：推荐 zap 结构化日志，Prometheus 指标可覆盖 WS 连接数、出价结果、Lua 延迟、落锤结果、广播丢弃、异步写库积压。
7. 配置建议：使用 `configs/config.yaml` + 环境变量覆盖，至少包含 server、mysql、redis、jwt、auction、websocket、observability 配置。
8. 本地依赖启动顺序：MySQL → Redis → 执行迁移/导入 DDL → 后端加载配置与 Lua 脚本 → 启动 HTTP/WS/调度器。

## 4. 数据库与模型开发步骤

1. 先落地迁移基线：以 `docs/ddl.sql` 为 MVP 初始 DDL，转为 `migrations/001_init_schema.sql` 或等价迁移文件。
2. 核心表必须覆盖：
   - `user`：买家、商家、管理员统一用户表。
   - `item`：商家商品。
   - `auction_lot`：拍品/拍卖实例，包含起拍价、保留价、规则快照、状态、开拍/结束时间、赢家和成交价。
   - `bid_record`：出价事实表，`request_id` 唯一保障幂等。
   - `order_deal`：成交订单，`auction_id` 唯一保障一拍一单。
   - `deposit_ledger`：保证金账本，`(auction_id,user_id)` 唯一保障同拍场单用户一条有效记录。
   - `audit_log`、`blacklist`、`risk_event`、`config_item`。
3. 金额模型统一：字段如 `start_price`、`reserve_price`、`deposit_amount`、`bid_price`、`deal_price`、`amount` 全部为「分」单位整数。
4. 状态机先在 `domain` 中定义，再映射到数据库字段：
   - 拍品：`DRAFT -> PENDING_AUDIT -> READY -> WARMING_UP -> RUNNING/EXTENDED -> HAMMER_PENDING -> CLOSED_WON/CLOSED_FAILED -> SETTLED`。
   - 订单：`CREATED -> PAID/TIMEOUT/CANCELLED`。
   - 保证金：`PENDING -> READY -> CAPTURED/RELEASED`，失败进入 `FAILED`。
5. 先实现 Repository 的基础 CRUD 和事务辅助，再实现 Service；落锤事务要在一个 MySQL 事务中生成订单、写保证金扣转/释放、更新拍品关闭状态。
6. 必须保留核心索引：
   - `auction_lot.idx_status_end_time(status,end_time)`：调度器扫描到期拍卖。
   - `bid_record.uk_request_id(request_id)`：出价幂等。
   - `bid_record.idx_auction_bid_ts(auction_id,bid_ts_ms)`：拍场回放。
   - `order_deal.uk_auction_id(auction_id)`：一拍一单。
   - `deposit_ledger.uk_auction_user(auction_id,user_id)`：保证金幂等。

## 5. Redis 与热路径开发步骤

Redis 是竞拍热路径权威缓存，MySQL 是最终事实表。出价、倒计时、幂等、频控、排行榜和落锤锁优先通过 Redis/Lua 原子处理。

| Key | 类型 | 用途 |
| --- | --- | --- |
| `auction:{id}:state` | Hash | 当前价、领先者、状态、开始/结束时间、版本、延时次数 |
| `auction:{id}:bids` | ZSet | 排行榜，建议 `score = price * 1e7 - ts_ms` |
| `auction:{id}:user_bids` | Hash | 每个用户最高价与出价时间 |
| `auction:{id}:idem:{request_id}` | String | 出价幂等，TTL 约 300 秒 |
| `auction:{id}:lock:close` | String | 落锤关闭锁，`SET NX EX/PX` |
| `auction:{id}:stream` | Stream | 竞拍事件流，用于补偿与断线恢复 |
| `online:auction:{id}` | Set/Hash | 在线用户或连接统计 |
| `risk:freq:bid:{user_id}:{auction_id}` | String | 出价频控计数 |
| `risk:blacklist:user` | Set | 黑名单缓存 |

开发步骤：

1. 封装 Redis Client 初始化、Ping 检查、连接池参数和关闭逻辑。
2. 封装 Lua Script Registry：启动时 `SCRIPT LOAD`，业务调用走 `EVALSHA`，遇到 `NOSCRIPT` 时 reload 并重试一次。
3. 编写 `bid.lua`：
   - 校验拍卖状态为运行中。
   - 校验用户已报名、保证金已冻结、不在黑名单。
   - 校验 `requestId` 幂等；重复请求返回上次结果，不推进状态。
   - 校验出价金额满足当前价 + 最低加价幅度。
   - 做频控计数，超限生成风险拒绝结果。
   - 原子更新最高价、领先者、排行榜、用户最高价、事件流。
   - 若命中 anti-sniping 窗口，原子延长 `end_ts_ms` 并返回 `timer.extended` 所需数据。
4. 编写 `hammer.lua`：
   - 通过 `auction:{id}:lock:close` 获取关闭锁。
   - 非强制落锤时校验当前时间已到结束时间。
   - 状态只能从运行中迁移到关闭中/已关闭。
   - 返回赢家、成交价、版本号、关闭结果，供 MySQL 事务落事实表。
5. 实现异步 `bid_record` 写库 worker：成功出价先以 Redis 结果为准，再异步落 MySQL；失败时记录日志和指标，并可通过 Redis Stream 补偿。

## 6. 核心模块开发步骤

### 6.1 鉴权与用户

1. 实现登录、登出、刷新令牌、当前用户信息接口。
2. JWT claims 至少包含 `sub`、`role`、`exp`、`iat`。
3. Hertz 中间件解析 JWT，将用户 ID、角色写入 context。
4. 角色不满足时返回 `403` 和业务错误码 `10003`。

### 6.2 商品模块

1. 实现商家商品 CRUD：创建、列表、详情、更新、删除。
2. 商家只能操作自己的商品；管理员可按后台接口跨商家管理。
3. 已绑定进行中拍品的商品不允许删除或修改关键字段。

### 6.3 拍品与规则模块

1. 实现拍品创建、查询、更新、取消、我的拍品。
2. 创建拍品时校验商品归属、起拍价 `>= 0`、保证金和加价规则合法。
3. 固化 `rule_snapshot`，后续规则调整不影响已创建拍品。
4. 明确状态迁移条件：草稿、待审核、就绪、运行中、已关闭、已取消。

### 6.4 报名与保证金

1. 用户通过 `/api/v1/auctions/{id}/enroll` 报名并模拟冻结保证金。
2. 报名前校验用户状态、黑名单、拍品状态和保证金策略。
3. `deposit_ledger` 写入 `PENDING/READY` 状态，重复报名按幂等返回。
4. 未中拍释放保证金；中拍后按业务配置扣转或抵扣尾款。

### 6.5 WebSocket Hub

1. 使用 Hertz WebSocket 提供连接入口，连接时校验 JWT、用户状态、拍品可见权限。
2. Hub 维护 `auction_id -> Room`，Room 内部串行处理注册、离开、广播、上行消息。
3. 每个 Client 使用 `readPump` 和 `writePump`，设置读写超时、ping/pong、发送 buffer。
4. 慢消费者不阻塞房间广播，`send` channel 满时主动断开连接并打点。
5. 重连时支持携带 `lastSeq`，并通过状态快照与事件流补偿缺失消息。

### 6.6 出价与排名

1. WS 上行 `bid.place` 必须带 `requestId`、`auctionId`、`bidPrice`、`clientTsMs`。
2. `BidService.Place` 只做参数和上下文准备，核心状态判断交给 `bid.lua`。
3. 成功后发送 `bid.ack`，广播 `bid.accepted`、`ranking.updated`，必要时发送 `bidder.overtaken` 与 `timer.extended`。
4. 失败后发送 `bid.rejected`，错误码对齐 API 文档，如 `50001 BID_TOO_LOW`、`50004 BID_RATE_LIMITED`。

### 6.7 倒计时与落锤

1. Room ticker 每秒读取 Redis 权威 `end_ts_ms/status` 并推送 `timer.tick`。
2. 到点由 Room ticker 或 cron 兜底触发 `HammerService`。
3. 手工落锤走 `/api/v1/auctions/{id}/hammer`，必须携带 `Idempotency-Key`。
4. `HammerService` 先执行 `hammer.lua`，再在 MySQL 事务中生成订单、扣转保证金、更新拍品状态。
5. 成功后广播 `auction.closed` 和 `order.created`；重复落锤返回已关闭快照，不重复生成订单。

### 6.8 风控与审计

1. 黑名单从 MySQL 回灌到 Redis `risk:blacklist:user`，报名和出价前均校验。
2. 出价频控按单用户单房间窗口计数，默认出价 ≤ 10 QPS。
3. 高风险命中写 `risk_event`，并可通过 `risk.event` 推送到 admin 房间。
4. 管理员审核、强制取消/关闭、黑名单变更、风控规则修改、落锤等关键操作写 `audit_log`。

### 6.9 订单与支付模拟

1. 落锤成交后创建 `order_deal`，订单初始为待支付/未支付。
2. 买家支付接口 `/api/v1/orders/{id}/pay` 使用幂等键，支付成功后更新 `PAID` 与 `paid_at`。
3. 支付超时、取消、退款等作为后续状态流转保留扩展点。

## 7. API 开发步骤

1. 先实现通用响应与错误码：所有接口返回 `{code,message,data,trace_id}`，业务错误码与 `docs/API文档.md` 对齐。
2. 实现中间件：RequestID、Recovery、JWTAuth、RoleAuth、Idempotency-Key、基础限流、审计埋点。
3. 按模块开发 REST：
   - Auth：`POST /auth/login`、`POST /auth/logout`、`POST /auth/refresh`、`GET /auth/me`。
   - Item：`POST/GET /items`、`GET/PATCH/DELETE /items/{id}`。
   - Auction：`POST/GET /auctions`、`GET/PATCH /auctions/{id}`、`POST /start`、`POST /hammer`、`POST /cancel`、`GET /state`、`POST /enroll`、`GET /auctions/mine`。
   - Order：`GET /orders`、`GET /orders/mine`、`GET /orders/{id}`、`POST /orders/{id}/pay`。
   - Admin：审核、强制取消/关闭、用户、黑名单、订单、审计日志、风控规则、风险事件、看板指标、配置。
   - AI 加分：`POST /ai/start-price-suggest`、`POST /ai/announcement`。
4. WebSocket 事件按 API 文档实现：
   - 上行：`bid.place`、`room.subscribe`、`room.unsubscribe`、`heartbeat`。
   - 下行：`bid.ack`、`bid.accepted`、`bid.rejected`、`ranking.updated`、`bidder.overtaken`、`timer.tick`、`timer.extended`、`auction.started`、`auction.closed`、`order.created`、`announcement`、`risk.event`、`error`。
5. API 版本兼容：`/api/v1` 内只做兼容新增；删除字段、字段语义改变、必填字段新增等破坏性变更必须升级版本。

## 8. 实时竞拍流程

完整竞拍闭环建议按以下顺序实现和联调：

1. 商家登录并创建商品。
2. 商家创建拍品，设置起拍价、最低加价、保证金、开始时间、持续时间、反狙击规则快照。
3. 管理员审核通过后拍品进入 `READY`。
4. 买家登录并报名拍品，保证金账本冻结成功。
5. 商家调用开拍接口：
   - MySQL 状态迁移为运行中。
   - Redis 初始化 `auction:{id}:state`、排行榜、倒计时字段。
   - WS 广播 `auction.started`。
6. 买家建立 WebSocket 连接并订阅房间。
7. 买家发送 `bid.place`：
   - Lua 原子校验状态、幂等、报名、黑名单、频控、最低加价。
   - 接受后更新 Redis 当前价、领先者、排行榜、事件流。
   - 命中反狙击窗口时延长 `end_ts_ms`。
   - 服务端发送 ACK 并广播排名、价格、延时事件。
8. 倒计时到点或商家手工落锤：
   - `hammer.lua` 通过关闭锁保证唯一关闭。
   - MySQL 事务生成订单、扣转保证金、更新拍品状态。
   - WS 广播 `auction.closed`、`order.created`。
9. 买家查询订单并支付；商家和后台查询订单及审计记录。
10. 客户端断线重连后调用 `GET /api/v1/auctions/{id}/state` 拉取快照，并根据 `lastSeq` 做事件补偿。

## 9. 测试与联调

### 9.1 单元测试

1. Lua 脚本必须覆盖：正常出价、低价拒绝、重复 `requestId`、黑名单、频控、反狙击延时、非运行状态拒绝、落锤重复执行。
2. Service 层覆盖：状态机迁移、权限校验、幂等冲突、事务回滚、唯一键冲突按幂等成功处理。
3. Repository 层覆盖关键索引查询、分页与事务。

### 9.2 集成测试

1. 使用本地 MySQL + Redis 导入 DDL 和种子数据。
2. 按「登录 → 创建商品 → 创建拍品 → 审核 → 报名 → 开拍 → WS 出价 → 落锤 → 订单支付」跑通主链路。
3. 验证重复出价、重复落锤、重复支付不会产生重复事实记录。
4. 验证 Redis 重启、应用重启后可从 MySQL 恢复运行中拍卖热状态。

### 9.3 联调检查

1. REST 用 `curl` 或 Postman 验证请求/响应字段与 API 文档一致。
2. WebSocket 用 `wscat`/`websocat` 验证事件信封、ACK、广播、心跳、断线重连。
3. 前端联调重点确认：金额单位、时间戳毫秒、枚举值、错误码、`requestId`/`Idempotency-Key`、WS 重连恢复。
4. 后台联调重点确认：审核、强制取消/关闭、黑名单、风控事件、审计日志、看板指标。

### 9.4 压测与一致性验收

1. 使用 k6、vegeta 或自研脚本模拟单房间 `100 ~ 500 QPS` 出价。
2. 观测出价 P99 是否 `< 300ms`，榜单广播延迟是否 `< 500ms`。
3. 压测后核对 Redis 最高价、MySQL 最新出价、订单成交价一致。
4. 并发触发多次自动/手工落锤，验证 `order_deal.auction_id` 唯一且仅生成一单。

## 10. 部署与验收

1. 准备 Dockerfile：多阶段构建 Go 二进制，运行镜像包含配置和 Lua 脚本。
2. 准备 docker-compose：至少包含 MySQL 8、Redis 7、server，并挂载迁移目录或初始化 SQL。
3. 服务启动流程：
   - 加载配置。
   - 初始化 logger、MySQL、Redis。
   - 加载 Lua 脚本并缓存 SHA。
   - 恢复运行中拍卖热状态。
   - 注册 REST 路由和 WebSocket 入口。
   - 启动 cron、Hub、Hertz Server。
4. 健康检查：
   - `/healthz`：进程存活。
   - `/readyz`：MySQL、Redis、Lua SHA、关键 worker 可用。
   - `/metrics`：Prometheus 指标。
5. 优雅关闭：停止接收新请求和新 WS 连接，关闭房间，等待异步写库 worker flush，关闭 Redis/MySQL。
6. 验收标准：
   - B-01 ~ B-15 必做功能全部可演示。
   - 出价 P99 `< 300ms`，WS 送达率目标 `> 99.9%`。
   - 落锤 100% 唯一，重复请求 100% 幂等。
   - 单房间 `100 ~ 500 QPS` 稳定运行。
   - Redis/MySQL/应用重启可恢复且不丢失成交记录。

## 11. 推荐开发顺序

### 第 1 阶段：基础框架与数据层

1. 工程目录、配置、日志、Hertz 启动与健康检查。

3. Redis Client、Lua Script Registry、基础 Key 封装。
4. JWT 鉴权、中间件、统一响应、错误码。
5. 商品 CRUD、拍品 CRUD、规则快照、状态恢复接口。
6. WebSocket Hub 骨架、房间管理、心跳、订阅与重连入口。

### 第 2 阶段：实时竞拍主链路

1. 开拍接口与 Redis 状态预热。
2. `bid.lua` 与出价 Service。
3. 排名广播、被超越定向推送、倒计时同步。
4. `hammer.lua` 与落锤 Service。
5. 报名保证金、订单生成、支付模拟。
6. 风控基础、黑名单、频控、审计日志。
7. 主链路集成测试与压测。

### 第 3 阶段：管理、加分与稳定性

1. 管理后台接口：审核、强制取消/关闭、用户、黑名单、订单、审计、风控规则。
2. 数据看板指标与 admin 监控房间。
3. AI 起拍价建议、直播公告文案等加分接口。
4. 重启恢复、补偿任务、Redis/MySQL 故障演练。
5. 端到端联调、部署脚本、验收材料。

## 12. 待确认事项

1. Go 版本是否以当前 `go.mod` 的 `1.26.2` 为准，还是按设计文档统一到 Go `1.22+` 的具体版本。
2. ORM 最终选型：GORM、sqlc 或 sqlx；若使用 GORM，需统一模型命名、表名映射和事务封装规范。
3. API 角色枚举需要统一：文档中同时出现 `user/buyer/merchant/admin`，需要确认买家角色最终使用 `user` 还是 `buyer`。
4. 拍品状态枚举需要统一：API 文档使用 `DRAFT/READY/LIVE/CLOSED/CANCELED`，数据库设计使用更细的 `RUNNING/EXTENDED/CLOSED_WON/CLOSED_FAILED/SETTLED`，实现前需确定对外枚举与内部枚举映射。
5. WebSocket 连接路径需要统一：API 文档示例为 `/ws?token=&auctionId=`，技术设计为 `/ws/auctions/{auction_id}?token=&lastSeq=`。
6. 出价请求字段需要统一：API 文档使用 `bidPrice`，技术设计示例使用 `amountCent`；建议对外使用 `bidPrice`，内部统一转换为 `AmountCent`。
7. 支付与保证金是纯模拟还是需要接真实/沙箱支付通道；若接外部支付，需要补充回调和对账设计。
8. 图片上传由后端本地存储、对象存储还是仅保存 URL；MVP 是否需要真正上传接口。
9. Redis 部署形态：MVP 单实例 + AOF 是否足够，是否需要 Sentinel/Cluster 的预留配置。
10. AI 能力接入范围：起拍价建议、公告生成、TTS 至少选择哪些作为第 3 周加分项，失败降级策略如何展示给前端。
