# 出价竞价校验演进设计

本文前半部分先只讨论拍卖出价过程中的价格校验问题，并且是在不考虑限流、黑名单校验的情况下推导规则。保证金、订单生成、反狙击延时等其他链路不在本文展开。

文末会把价格校验和限流、黑名单策略合并，形成最终可落地的完整出价防护方案。

## 背景

当前拍卖采用固定加价规则：

```json
{
  "amount": 10,
  "maxBidSteps": 3
}
```

含义是：

- 加价步长：固定加价的最小单位。
- 单次最多加价步数：一次出价最多允许跨多少个步长。
- 封顶价：出价达到该价格后自动成交。

例如当前价是 `100`，步长是 `10`，单次最多 `3` 步，那么基于当前价，本次最高普通出价是：

```text
100 + 10 * 3 = 130
```

如果封顶价是 `140`，那么出到 `140` 会触发自动成交。

## 第一版设计：只基于 Redis 最新价格校验

最初设计是：

1. Go 后端先做一次预校验。
2. Redis Lua 再做一次最终校验。
3. Lua 以 Redis 中最新的当前价为准。
4. Lua 通过后原子更新当前价、领先者、版本号和出价流。

核心规则是：

```text
本次出价 > 当前价
本次出价 <= 封顶价
普通出价满足 (本次出价 - 起拍价) % 加价步长 == 0
本次出价 <= min(封顶价, 当前价 + 加价步长 * 单次最多加价步数)
```

这个设计解决了并发一致性问题。多个出价同时到达时，Redis Lua 是串行执行的，后一个请求一定基于前一个请求更新后的最新价格重新判断。

## 问题一：WebSocket 不是安全边界

出价虽然是 WebSocket 消息，但 WebSocket 不是“只能前端页面操作”的保护。

脚本同样可以建立 WebSocket 连接，然后持续发送：

```json
{
  "type": "bid.place",
  "requestId": "req-xxx",
  "payload": {
    "auctionId": 1,
    "price": 140
  }
}
```

只要脚本有有效登录态，并且满足报名、保证金等条件，它和正常前端按钮发出的消息没有本质区别。

因此，防抢单不能依赖前端禁用按钮、前端价格限制，也不能依赖 WebSocket 本身。最终规则必须落在 Go 后端和 Redis Lua 中，尤其是 Lua 最终校验。

## 问题二：提前挂封顶价抢单

假设：

```text
起拍价 = 100
当前价 = 100
加价步长 = 10
单次最多加价步数 = 3
封顶价 = 140
```

脚本一直发 `本次出价=140`。

当 Redis 当前价还是 `100` 时：

```text
本次最高允许价 = min(140, 100 + 10 * 3) = 130
本次出价 = 140
```

此时 Lua 会拒绝，因为 `140 > 130`。

但如果另一个正常用户先出价 `110`，Redis 当前价变成 `110`，脚本下一次 `140` 进入 Lua 时：

```text
本次最高允许价 = min(140, 110 + 10 * 3) = 140
本次出价 = 140
```

此时基于 Redis 最新价格，`140` 是合法的，并且正好等于封顶价，于是自动成交。

问题在于：脚本发出 `140` 时，并不是基于它看到 `110` 后做出的用户确认，而是提前挂着请求等待别人把价格垫上来。

## 尝试方案一：要求 用户看到的当前价 等于 Redis 当前价

一种直接想法是在出价请求里增加：

```json
{
  "price": 140,
  "expectedCurrentPrice": 100
}
```

Lua 校验：

```text
用户看到的当前价 == Redis 最新当前价
```

这样可以挡住提前挂封顶价。

脚本在看到 `100` 时发 `140`，等 Redis 当前价变成 `110` 后再执行，Lua 会发现：

```text
用户看到的当前价 = 100
Redis 最新当前价 = 110
```

两者不一致，直接拒绝。

但这个方案过于严格。

## 问题三：严格相等会误伤正常出价

假设正常用户看到当前价 `100`，手动出价 `120`。

这个请求发出后，另一个用户的 `110` 先到 Redis 并成功。

此时用户的 `120` 请求进入 Lua：

```text
用户看到的当前价 = 100
Redis 最新当前价 = 110
本次出价 = 120
```

如果要求 `用户看到的当前价 == Redis 最新当前价`，那么 `120` 会被拒绝。

但从业务上看，`120` 是一个合理出价：

```text
120 >= 110 + 10
```

也就是说，严格相等可以防抢单，但会降低正常竞价成功率。它把所有“基于旧价格发出的请求”都判定为过期，即使这个价格在最新状态下仍然合法。

## 尝试方案二：只要求 用户看到的当前价 小于等于 Redis 当前价

另一种想法是：

```text
用户看到的当前价 <= Redis 最新当前价
```

这样正常用户看到 `100` 出 `120`，即使别人先出到 `110`，也不会因为价格变化被拒绝：

```text
100 <= 110
```

但这个条件本身挡不住提前挂封顶价。

还是前面的攻击场景：

```text
用户看到的当前价 = 100
Redis 最新当前价 = 110
本次出价 = 140
封顶价 = 140
```

判断结果是：

```text
用户看到的当前价 <= Redis 最新当前价
100 <= 110 成立
```

再套原来的 Redis 最新价规则：

```text
本次最高允许价 = min(140, 110 + 10 * 3) = 140
本次出价 = 140 成立
```

所以它仍然会自动成交。

结论是：`用户看到的当前价 <= Redis 最新当前价` 可以作为基础一致性检查，但不能作为核心防抢单规则。

## 最终方案：双基准最高价校验

最终方案由三部分组成：双基准最高价校验、同用户限流、明显异常价格黑名单。双基准最高价校验负责判断“这笔价格是否合法”，限流负责拦截高频脚本请求，黑名单负责处理明显不可能由正常前端操作产生的异常价格。

第一，价格校验不要求用户看到的价格必须等于 Redis 最新价格，而是同时使用两个基准价格：

- 用户看到的当前价：用户发起出价时页面展示的当前价。
- Redis 最新当前价：Lua 执行瞬间 Redis 中保存的当前价。

出价必须同时满足：

```text
用户看到的当前价 <= Redis 最新当前价
本次出价 <= min(封顶价, 用户看到的当前价 + 加价步长 * 单次最多加价步数)
本次出价 <= min(封顶价, Redis 最新当前价 + 加价步长 * 单次最多加价步数)
```

其中：

- 第 1 条保证客户端不是基于一个“未来价格”构造请求。
- 第 2 条保证本次出价没有超过用户看到价格时允许的最高出价。
- 第 3 条保留原有 Redis 最新状态校验，保证并发下仍以最新价格为准。

再叠加原有规则：

```text
本次出价 > 起拍价
本次出价 > Redis 最新当前价
本次出价 <= 封顶价
普通出价满足 (本次出价 - 起拍价) % 加价步长 == 0
本次出价 == 封顶价 时允许不按步长对齐
```

第二，同时对同一用户加限流，避免脚本高频请求持续消耗后端、Redis 和 WebSocket 资源。限流可以至少分两层：

```text
同一用户 + 同一拍品：短窗口限流
同一用户 + 全站出价：全局限流
```

例如同一用户同一拍品 `1` 秒最多 `3` 次出价请求，同一用户全站 `10` 秒最多 `20` 次出价请求。超过限制时直接拒绝；如果短时间内大量触发限流，可以记录风险事件，并进入更严格的风控策略。

第三，对明显不合理的价格请求加黑名单策略。比如当前价 `100`、最多只能出到 `130`，用户却持续请求 `140`，这种请求不可能通过正常前端按钮产生，可以判定为明显脚本行为，直接加入黑名单或立即禁止其继续参与当前拍品。

黑名单生效后，Lua 在出价前先检查用户是否在黑名单中；如果命中黑名单，直接拒绝。这样可以避免脚本持续用明显不合理的价格撞库，等待别人把价格垫上来后抢封顶价。

## 正常出价示例

假设：

```text
用户看到的当前价 = 100
Redis 最新当前价 = 110
本次出价 = 120
加价步长 = 10
单次最多加价步数 = 3
封顶价 = 140
```

校验：

```text
用户看到的当前价 <= Redis 最新当前价
100 <= 110 成立

用户视角最高允许价 = min(140, 100 + 10 * 3) = 130
Redis 视角最高允许价 = min(140, 110 + 10 * 3) = 140

本次出价 = 120
120 <= 130 成立
120 <= 140 成立
120 > 110 成立
```

结果：通过。

这个方案不会误伤“看到 100 出 120，但别人先出到 110”的正常请求。

## 抢封顶价脚本示例

假设脚本一直发：

```text
用户看到的当前价 = 100
本次出价 = 140
```

别人先出价后，Redis 当前价变成：

```text
Redis 最新当前价 = 110
```

校验：

```text
用户看到的当前价 <= Redis 最新当前价
100 <= 110 成立

用户视角最高允许价 = min(140, 100 + 10 * 3) = 130
Redis 视角最高允许价 = min(140, 110 + 10 * 3) = 140

本次出价 = 140
140 <= 130 不成立
```

结果：拒绝。

拒绝原因可以定义为：

```text
ABOVE_EXPECTED_MAX_BID_STEPS
```

含义是：本次出价超过了用户发起请求时看到价格所允许的最高出价。

## 用户刷新后确认封顶价

如果用户确实看到最新价 `110`，并明确要出封顶价 `140`，请求应该变成：

```text
用户看到的当前价 = 110
本次出价 = 140
```

校验：

```text
用户视角最高允许价 = min(140, 110 + 10 * 3) = 140
Redis 视角最高允许价 = min(140, 110 + 10 * 3) = 140
本次出价 = 140
```

结果：通过，并触发封顶成交。

这符合业务预期：封顶成交必须来自用户基于最新价格区间做出的确认，而不是提前挂单等待别人垫价。

## Go 与 Lua 的职责划分

Go 后端仍然做预校验：

- 参数格式。
- 用户角色。
- 拍品存在。
- 出价规则解析。
- 基于当前可读取状态做快速失败。

但 Go 预校验不能作为最终依据，因为 Go 校验后到 Lua 执行前，Redis 状态可能已经变化。

Redis Lua 必须做最终校验：

- 读取 Redis 中最新的当前价。
- 读取最新版本号、领先者、拍卖状态。
- 校验用户看到的当前价与双基准最高价。
- 校验同用户出价频率。
- 校验用户是否已经进入黑名单。
- 对明显异常价格记录风险，并按策略加入黑名单或禁止继续参与当前拍品。
- 原子写入最新价格、领先者、版本和出价事件。

前端的限制只是体验优化，不能作为安全规则。
