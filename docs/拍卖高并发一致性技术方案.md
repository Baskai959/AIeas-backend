# 拍卖高并发一致性技术方案

## 1. 背景与目标

当前系统是实时拍卖/直播间后端，核心链路包括报名、保证金、出价、落槌、订单生成、出价记录落库和 WebSocket 广播。后续如果进行多实例部署，需要同时满足以下目标：

- 出价高并发下仍能保证最高价、领先者、倒计时延时、排名一致。
- 多实例部署下任意后端实例收到出价，都能让所有 WebSocket 客户端尽快看到同一裁决结果。
- MySQL 不参与实时裁决，避免数据库成为出价热路径瓶颈。
- 出价记录、成交订单、保证金状态最终可靠落库，并支持幂等重试和对账补偿。
- 在 Redis、Kafka、MySQL、WebSocket 任一环节局部故障时，系统有明确的恢复边界。

总体原则：

```text
实时裁决走 Redis Lua
事件日志驱动广播与落库
MySQL 保存最终事实
跨存储不做分布式事务，靠幂等、唯一键、事件重放和对账补偿保证最终一致
```

## 2. 当前实现概览

当前代码已经具备较好的基础：

- `BidService.Place` 负责出价入口校验，最终调用 `AuctionRealtimeStore.PlaceBid`。
- `scripts/lua/bid.lua` 在 Redis 内原子完成出价幂等、状态校验、报名/保证金校验、黑名单校验、频控、最小加价、延时、排名更新、事件写入。
- `scripts/lua/hammer.lua` 在 Redis 内原子完成落槌实时态关闭和最高价读取。
- `HammerService.Hammer` 在 MySQL 事务内更新拍品终态、创建订单、捕获赢家保证金、释放其他保证金。
- `bid_record` 表通过 `uk_request_id` 保证出价记录幂等。
- `order_deal` 表通过 `uk_auction_id` 保证一拍一单。
- `deposit_ledger` 表通过 `uk_auction_user` 保证同一拍品同一用户只有一笔保证金账本。
- Redis Stream 已用于出价事件记录，后台 `BidRecordWriter` / `BidRecordReconciler` 负责异步落库与补偿。

需要优化的点：

- 出价热路径不应先查 MySQL 出价记录幂等。MySQL 查询会成为高并发瓶颈，也可能因为异步落库延迟查不到最新记录。
- WebSocket 多实例广播不能只依赖当前处理出价的实例本机 Hub。需要跨实例事件 fan-out。
- Redis Stream 如果通过轮询广播，会引入轮询延迟。低延迟广播建议引入 Pub/Sub 或 Kafka 实时 topic，同时保留 Stream/Kafka 作为可靠日志。

## 3. 推荐目标架构

推荐链路：

```text
客户端出价
  -> 任意后端实例
  -> Go 参数校验和可选预过滤
  -> Redis master 执行 Lua 原子裁决
       - 幂等检查
       - 用户限流
       - 拍卖状态校验
       - 报名/保证金校验
       - 当前价和最小加价校验
       - 更新最高价/领先者/排名/倒计时
       - 写 Redis 幂等结果
       - 写 Redis Stream 可靠事件
       - 可选 PUBLISH 低延迟广播事件
  -> 出价接口返回 Lua result

广播链路
  -> 所有 WebSocket 实例订阅实时广播事件
  -> 每个实例只广播给本机连接的客户端

落库链路
  -> Redis Stream -> MySQL 幂等批量落库
  或 Redis Stream -> Kafka -> MySQL 幂等批量落库
```

关键边界：

- **最新价格来源**：出价广播使用 Lua result / Lua 写出的事件 payload，不在广播时二次查询 Redis。
- **Redis state 用途**：用户刚连接时读取快照、断线重连补状态、HTTP 查询当前态、定时落槌读取状态。
- **MySQL 用途**：保存最终事实、历史查询、审计、订单和保证金账本，不参与出价实时裁决。

## 4. 幂等设计

### 4.1 HTTP 接口幂等

状态变更接口继续使用 `Idempotency-Key`。中间件按以下维度缓存成功响应：

```text
userId + method + path + idempotencyKey + requestBodyFingerprint
```

同一个幂等键重复请求返回缓存结果；同一个幂等键但请求体不同返回冲突。

适用范围：

- 启动拍卖
- 取消拍卖
- 落槌
- 直播间激活/下播
- 订单支付
- 管理后台状态变更

### 4.2 出价幂等

出价幂等主权应放在 Redis Lua 内：

```text
GET auction:{auctionId}:idem:{requestId}
  命中 -> 直接返回旧 result
  未命中 -> 继续裁决，裁决结束后 SET PX 保存 result
```

MySQL `bid_record.uk_request_id` 只作为异步落库的最终幂等约束，不应作为出价热路径第一查询。

### 4.3 落库幂等

出价事件可能重复消费，MySQL consumer 必须幂等：

- `bid_record.request_id` 唯一。
- 插入冲突时读取已有记录并校验是否一致。
- 一致则 ack。
- 不一致则写入 DLQ，人工或后台任务处理。

订单也必须幂等：

- `order_deal.auction_id` 唯一。
- 落槌重复执行时返回已有订单。

## 5. 高并发设计

### 5.1 出价热路径 Redis-only

出价热路径只访问 Redis master：

```text
Go 参数校验 -> Redis Lua -> 返回结果
```

Go 可以做预过滤，但只能作为减压手段，不能作为最终裁决：

- 参数缺失、价格 <= 0、角色不合法可以直接拒绝。
- 可选从 Redis master 读取当前价，挡掉明显低价请求。
- 如果从 Redis replica 读取当前价，只能做保守拒绝，不能判定请求有效。

最终有效性必须由 Lua 内基于最新实时态再次校验。

### 5.2 不建议出价分布式锁

多实例部署下不建议为每次出价加 Redis 分布式锁。

原因：

- Redis Lua 本身已经是原子串行裁决点。
- 加锁会把一次 Lua 操作变成 `SET NX -> Lua -> DEL`，增加 Redis 往返和失败点。
- 锁过期、续期、误删锁会引入额外一致性风险。
- 没抢到锁的高价请求如果直接失败，会破坏拍卖体验；如果重试，又增加尾延迟和流量。

分布式锁适合低频互斥场景：

- 同一直播间同时只能激活一个拍品。
- 后台任务抢占。
- 结算任务防重复执行。

### 5.3 Redis Cluster 分片

不同拍品之间可以水平扩展。Redis Cluster 下需要使用 hash tag，保证同一拍品相关 key 落在同一个 slot：

```text
auction:{10001}:state
auction:{10001}:bids
auction:{10001}:idem:{requestId}
auction:{10001}:enrolled
auction:{10001}:deposits
auction:{10001}:stream
auction:{10001}:freq:{userId}
```

这样单个拍品内 Lua 可原子执行，不同拍品分散到不同 Redis shard。

### 5.4 热门拍品隔离

单个超级热门拍品最终会受单 Redis shard 单线程限制。可选策略：

- 热门拍品提前分配到独立 Redis 实例或独立 shard。
- 普通拍品按 `auctionId` 均匀分片。
- 监控单拍品 Lua 耗时、Redis CPU、队列长度、WebSocket 广播延迟。
- 当单拍品 Lua 成为瓶颈时，再考虑独立 bidding worker / actor 模型。

## 6. 一致性设计

### 6.1 实时态一致性

实时态以 Redis master 为准，Lua 一次完成：

- 当前价格
- 领先者
- 排名
- 结束时间
- 延时次数
- 出价事件序号
- 请求幂等结果

同一拍品内所有关键 key 必须在同一个 Redis slot。

### 6.2 持久态最终一致

MySQL 不同步参与出价裁决，而是通过事件异步落库：

```text
Redis Lua 更新实时态 + 写事件
后台 worker 消费事件
MySQL 幂等写入
reconciler 定期按 seq 回放补账
```

这意味着 Redis 实时态和 MySQL 出价记录之间允许短暂延迟。业务查询实时价格应读 Redis，历史记录和审计读 MySQL。

### 6.3 启动拍卖与报名保证金一致性

启动拍卖、报名/保证金同步 Redis-RT 时，不建议按普通缓存处理，也不能只采用“改数据库删缓存”。Redis-RT 中的拍品 state、报名集合、保证金 ready 集合是出价 Lua 的裁决索引，不是可随时回源的展示缓存。

推荐采用“类 TCC/中间态 + 同步写 Redis-RT + 重试补偿”：

```text
启动拍卖：
  READY
    -> WARMING_UP/STARTING
    -> 初始化 Redis-RT auction state
    -> RUNNING

初始化 Redis-RT 失败：
  -> 接口返回可重试
  -> 状态回退到 READY 或保持 WARMING_UP 等待补偿
  -> 后台 reconciler 扫描并重建 Redis-RT state
```

如果不新增 `STARTING` 状态，可复用 `WARMING_UP` 作为中间态，但需要保证 `Start` 接口幂等：重复调用时能识别 MySQL 已进入启动流程，并补齐缺失的 Redis-RT state。

报名/保证金：

```text
MySQL transaction:
  -> deposit_ledger READY

事务提交后：
  -> SADD Redis-RT enrolled/deposits

Redis-RT 写失败：
  -> 接口返回可重试或提示稍后重试
  -> 后台 reconciler 扫描 deposit_ledger READY/CAPTURED 补 Redis-RT set
```

这里不要求完整 TCC 的 Cancel 语义。`deposit_ledger READY` 是权威事实，Redis-RT set 是裁决索引；索引写失败时优先重试和补偿，而不是回滚已经成功的报名事实。

### 6.4 落槌一致性

落槌分两阶段：

```text
Redis hammer Lua
  -> 原子关闭实时态
  -> 读取最高价/赢家
  -> 写 close result 幂等锁

MySQL transaction
  -> 更新 auction_lot 终态
  -> 创建 order_deal
  -> 捕获赢家保证金
  -> 释放其他保证金
```

如果 Redis 落槌成功但 MySQL 事务失败：

- 重试同一个落槌请求应从 Redis close result 或 DB 终态恢复。
- `order_deal.uk_auction_id` 防止重复订单。
- 保证金更新在 MySQL 事务内完成，避免订单和保证金账本不一致。

如果落槌需要通过 Kafka 削峰或提高下游可靠性，推荐事件链路为：

```text
Redis hammer Lua
  -> 原子关闭 Redis state
  -> XADD Redis Stream auction.closed

Bridge/Worker
  -> Redis Stream -> Kafka

Kafka consumer
  -> MySQL 幂等事务落库
```

不要采用“Redis 关闭成功后仅由 Go 进程直接发 Kafka”作为唯一事件来源，否则 Go 进程在发 Kafka 前崩溃会丢失关闭事件。Redis Stream 应作为 Redis 状态变化的本地可靠事件日志。

### 6.5 自动/人工落槌并发控制

自动落槌、商家人工落槌、管理员强制关闭可能同时发生。正确性基础不应依赖分布式锁，而应依赖状态机、version 乐观锁、唯一约束和幂等。

推荐 MySQL 侧使用条件更新：

```sql
UPDATE auction_lot
SET status = ?,
    winner_id = ?,
    deal_price = ?,
    closed_at = ?,
    closed_by = ?,
    version = version + 1
WHERE auction_id = ?
  AND version = ?
  AND status IN ('RUNNING', 'EXTENDED', 'HAMMER_PENDING');
```

语义：

- 状态机保证 `CLOSED_WON`、`CLOSED_FAILED`、`SETTLED` 这类终态不可逆。
- `version` 保证并发关闭只有一个请求能更新成功。
- `RowsAffected = 0` 时，重新读取当前拍品状态；如果已经终态，则返回已有终态结果。
- `order_deal.auction_id` 唯一约束保证一拍一单。
- 保证金捕获/释放必须幂等。
- Redis `close_lock_key` / close result 继续用于实时态关闭幂等。

分布式锁可以作为降并发优化，但不作为一致性正确性的唯一保障。即使锁过期、释放失败或重复请求进入，状态机 + version + 唯一约束也必须保证最终结果正确。

### 6.6 Redis Stream 可靠性边界

Redis Stream 可以保留消息、支持 consumer group、pending、claim 和重试，但可靠性取决于 Redis 配置：

- 开启 AOF。
- 关键实例使用 `noeviction`。
- 不对未消费的关键 Stream 做危险裁剪。
- 如需更高可靠性，可使用 `WAIT` 等待副本确认，但会增加延迟。
- 定期对账 Redis Stream seq 与 MySQL `bid_record`。

如果业务要求长期保留、多下游、高可靠消息总线，应引入 Kafka。

## 7. WebSocket Gateway 与稳定性优化

### 7.1 广播不要二次查 Redis

出价广播使用 Lua result 或 Lua 写出的事件 payload：

```json
{
  "auctionId": 10001,
  "currentPrice": 12000,
  "leaderBidderId": "u_1001",
  "endTsMs": 1710000000000,
  "extendCount": 1,
  "seq": 18,
  "event": "bid.accepted"
}
```

不要在广播时再查 Redis 当前价。二次查询会增加 RTT，也可能因为读从库导致旧数据。

### 7.2 独立 WebSocket Gateway

中期建议将 WebSocket 从业务后端拆出，建设独立 WebSocket Gateway / Realtime Gateway：

```text
客户端
  -> WebSocket Gateway 集群
       负责长连接、订阅关系、心跳、推送、慢客户端隔离

客户端出价
  -> API Backend 集群
       负责鉴权、业务校验、Redis Lua 裁决

Redis-RT / Kafka / PubSub
  -> 作为裁决事件总线
```

职责边界：

- API Backend 负责出价裁决、订单、保证金、管理后台等业务逻辑。
- WebSocket Gateway 负责连接管理和消息广播，不做出价裁决，不查 MySQL，不保存权威业务状态。
- Gateway 可以读取 Redis-RT 当前快照，用于连接初始化和断线恢复。
- Gateway 通过 Redis Pub/Sub、Kafka topic 或 Redis Stream 独立消费广播事件。

这样 API 服务发布、重启或扩容时，不会直接影响大量 WebSocket 长连接；WebSocket 连接容量也可以独立扩容。

### 7.3 多实例 fan-out

每个 Gateway 实例只管理本机 WebSocket 连接。跨实例广播需要事件总线：

```text
Redis Lua 写事件
  -> 所有 WebSocket Gateway 实例收到同一事件
  -> 各实例广播给本机客户端
```

可选实现：

- Redis Pub/Sub：低延迟广播，断线期间消息不可恢复。
- Redis Stream：可恢复，但如果轮询会有延迟；广播场景不能用同一个 consumer group 抢消息。
- Kafka topic：适合多实例 fan-out 和多下游；每类消费者使用独立 consumer group。

推荐组合：

```text
Lua XADD Redis Stream 作为可靠日志
Lua PUBLISH Redis Pub/Sub 作为低延迟 WebSocket 广播
后台 worker 从 Stream 或 Kafka 落库
```

### 7.4 心跳保活

Gateway 需要主动维护连接健康：

```text
server ping -> client pong
超过 N 次未响应 -> 主动断开并清理连接
```

建议：

- 心跳间隔：15s - 30s。
- 超时时间：2 - 3 个心跳周期。
- 断开连接时清理本机连接表、订阅关系和在线人数。
- 客户端长时间未收到服务端消息或 ping/pong 异常时，主动重连。
- 负载均衡 idle timeout 必须大于心跳间隔，避免健康连接被 LB 误断。

### 7.5 快照、断线重连与 seq 去重

客户端连接 WebSocket 时：

1. Gateway 鉴权并建立连接。
2. Gateway 从 Redis-RT state 读取当前快照。
3. 返回 `currentPrice`、`leaderBidderId`、`endTime`、`seq/version`。
4. 后续通过事件增量更新。

快照内容至少包括：

```json
{
  "auctionId": 10001,
  "currentPrice": 12000,
  "leaderBidderId": "u_1001",
  "endTime": "2026-05-25T10:00:00Z",
  "seq": 18,
  "status": "RUNNING"
}
```

客户端断线重连：

```text
断线
  -> 指数退避重连，例如 1s, 2s, 5s, 10s
  -> 重连成功后先拉当前快照
  -> 可选携带 lastSeq 请求补消息
  -> 之后继续接收增量事件
```

事件必须带单调递增 `seq`。客户端保存 `lastSeq`：

```text
event.seq <= lastSeq -> 丢弃
event.seq > lastSeq  -> 应用事件并更新 lastSeq
```

这样即使多实例广播出现乱序、重复投递或客户端重连，也不会用旧消息覆盖新状态。

### 7.6 统一倒计时

用户看到的倒计时应基于服务端统一 `endTime`，不能由各客户端按本地开始时间自行倒计时。权威时间源为 Redis-RT 的 `end_ts_ms`。

服务端下发 snapshot 和时间变更事件时，需要包含：

```json
{
  "auctionId": 10001,
  "endTime": "2026-05-25T10:00:00.000Z",
  "serverTime": "2026-05-25T09:59:30.000Z",
  "seq": 18
}
```

客户端按以下方式计算：

```text
serverOffset = serverTime - clientNow
adjustedNow = clientNow + serverOffset
remainMs = endTime - adjustedNow
```

服务端不建议每秒广播 `timer.tick`。更好的方式是：

- snapshot 下发当前 `endTime` 和 `serverTime`。
- 防狙击延时时，Lua 更新 Redis-RT `end_ts_ms` 并推 `timer.extended` / `bid.accepted`，事件中携带新的 `endTime`、`serverTime`、`seq`。
- 客户端本地渲染倒计时。
- 服务端只在 `endTime` 变化、拍卖关闭、客户端重连快照时修正时间。

所有最终判定仍以后端为准。客户端本地倒计时到 0 后进入“结束确认中”，等待服务端关闭事件或重新拉取 snapshot。

### 7.7 慢客户端隔离

慢客户端不能拖慢整个拍品广播。Gateway 每个连接都需要独立发送缓冲：

```text
auction event -> per-connection bounded queue -> socket writer
```

建议：

- 每个连接使用有界队列，防止内存无限增长。
- 队列满时优先丢弃可合并的非关键消息，例如 `timer.tick`、在线人数、排名摘要。
- `bid.accepted`、`timer.extended`、`auction.closed` 等关键消息不能随意丢弃；如果关键消息也长期无法发送，直接断开慢连接，让客户端通过重连快照恢复。
- 排名、在线人数、统计类消息做聚合和限频，例如 100ms - 1000ms 推送一次。

### 7.8 推送链路解耦

出价裁决不能同步等待 WebSocket 广播完成。链路应保持解耦：

```text
出价请求
  -> Redis Lua 裁决
  -> 返回 Lua result 给出价人
  -> 事件总线 fan-out
  -> WebSocket Gateway 异步广播
```

要求：

- WebSocket 推送失败不能影响出价成功。
- Gateway 不查 MySQL，不做复杂业务计算。
- Gateway 只消费裁决事件并按订阅关系广播。
- 推送事件 payload 直接来自 Lua result / event，不二次查询当前价。

### 7.9 最后一秒出价与客户端结束确认

最后一秒有人出价时，不能由客户端本地倒计时直接判定竞拍结束。客户端倒计时只用于展示，最终结束必须以服务端 Redis 实时态和落槌结果为准。

服务端规则：

- 出价 Lua 在原子操作内读取当前 `end_ts_ms`。
- 如果出价发生在防狙击窗口内，Lua 同时更新最高价、领先者、`end_ts_ms`、`extend_count`，并生成 `timer.extended` / `bid.accepted` 事件。
- 自动落槌任务读取 Redis state 的最新 `endTime`，只有当前时间超过最新 `endTime` 后才允许落槌。
- 落槌 Lua 会再次基于 Redis 当前状态做一次性关闭，防止重复落槌。

客户端规则：

```text
本地倒计时到 0
  -> 进入“结束确认中”状态
  -> 等待一个短暂确认窗口，例如 1000ms
  -> 如果期间收到 bid.accepted / timer.extended：
       使用事件中的 endTime 重置倒计时，继续竞拍
     如果收到 auction.closed：
       展示成交/流拍结果
     如果确认窗口后仍未收到新事件：
       主动拉取 Redis state 对应的当前拍品状态
       以服务端状态决定展示“已结束”还是继续倒计时
```

这个 1 秒等待窗口只是客户端体验层的防抖和补偿，不能替代服务端判定。服务端仍然需要保证：

- 最后一秒有效出价会由 Lua 原子延时。
- WebSocket 广播延迟不会影响服务端是否延时。
- 客户端没收到广播时，可以通过重新拉取状态修正展示。

### 7.10 延迟与稳定性优化点

- Pub/Sub 广播和 Stream 可靠日志并行写入 Lua。
- WebSocket Gateway 不查 MySQL，不做复杂计算。
- 按 `auctionId` 或 shard 维度维护连接索引，避免全量遍历。
- 热门拍品使用独立 broadcast channel 或 shard channel。
- 消息 payload 尽量小，避免大对象 JSON。
- `timer.tick` 尽量客户端本地计算，服务端只推 `endTime` 修正。
- API Backend 和 Gateway 分开扩容、分开发布。
- Gateway 滚动发布时先停止接收新连接，再通知旧连接重连或等待连接自然迁移。

### 7.11 WebSocket 监控指标

建议监控：

- 当前连接数、每个 Gateway 实例连接数。
- 每个拍品/直播间订阅人数。
- 连接建立速率、断开速率、重连率。
- 心跳超时数量。
- 每连接发送队列长度。
- 慢客户端断开数量。
- 单条事件从 Lua 生成到 Gateway 收到的延迟。
- 单条事件从 Gateway 收到到客户端确认/接收的延迟。
- Pub/Sub、Kafka 或 Stream 消费延迟。
- 每拍品广播 fan-out 耗时 P95/P99。

## 8. Redis 读写分离策略

Redis 读写分离可以提高展示读并发，但不应参与实时裁决。

必须走 master：

- 出价 Lua。
- 当前价裁决读取。
- 幂等 key。
- 单用户限流。
- 报名/保证金校验。
- 落槌 Lua。
- Stream 写入。
- 分布式锁。

可以走 replica：

- 非关键统计。
- 历史榜单。
- 已结束拍卖展示。
- 拍品详情缓存。
- 可接受短暂延迟的展示型状态。

如果 Go 端预校验从 replica 读当前价，只能做保守拒绝：

```text
replica 当前价已经高于用户出价 -> 可拒绝
replica 当前价低于用户出价 -> 不能判定有效，仍需 Lua 裁决
```

## 9. Redis 分层缓存设计

缓存需要按数据一致性要求分层，不能把所有读都做成本地缓存。原则是：**裁决态不进本地缓存，展示态允许短暂不一致**。

### 9.1 Redis 实例按职责拆分

生产环境建议至少拆成两套 Redis：

```text
Redis-RT：实时竞拍 Redis
Redis-Cache：通用缓存 Redis
```

`Redis-RT` 只服务出价裁决和实时状态：

- 拍品实时状态、出价 Lua、幂等、限流、报名/保证金 ready、排名、落槌 close lock、直播间 active lock、竞拍事件 Stream/PubSub。
- 使用 `noeviction`，不允许因为内存压力淘汰关键竞拍 key。
- 关键实时状态不按普通缓存 TTL 过期；只允许限流、幂等、锁这类临时技术 key 使用明确 TTL。
- 低延迟优先，避免被商品缓存、大 key、缓存雪崩流量污染。

`Redis-Cache` 只服务允许短暂不一致的展示型缓存：

- 商品详情、直播间基础信息、商家信息、拍品展示配置、已结束拍卖结果、非强一致统计、空值缓存、Bloom Filter。
- 可以使用读写分离。
- 淘汰策略建议使用 `allkeys-lru`，在缓存内存达到上限时优先淘汰最近最少使用的 key。
- 所有业务缓存 key 必须设置 TTL，并添加随机抖动。

### 9.2 不允许本地缓存的数据

以下数据会影响出价裁决或资金/订单结果，必须读 Redis master 或 MySQL 权威源：

- 拍品实时状态：`currentPrice`、`leaderBidderId`、`endTime`、`extendCount`。
- 出价幂等 key。
- 用户限流 key。
- 报名和保证金 ready 集合。
- 黑名单裁决数据。
- 落槌 close lock。
- 订单支付状态和保证金账本状态。

这些数据可以使用 Redis，但不能使用各 Go 实例本地缓存参与判断，否则多实例下会出现过期读和错误裁决。

### 9.3 允许本地缓存的数据

以下数据可以使用本地缓存，允许秒级短暂不一致：

- 商品基础信息：标题、描述、图片、类目、规格。
- 拍品展示信息：起拍价、保证金金额、加价规则快照、开拍时间。
- 直播间基础信息：标题、封面、商家信息。
- 已结束拍卖结果展示。
- 非强一致统计：浏览量、在线人数展示值、历史榜单摘要。

推荐层次：

```text
本地进程缓存
  -> Redis 缓存
  -> MySQL
```

本地缓存用于减少 Redis 热读压力，Redis 缓存用于减少 MySQL 读压力。

### 9.4 缓存 TTL 与失效策略

建议策略：

- 商品/直播间基础信息本地缓存 TTL：1s - 5s。
- 已结束拍卖结果本地缓存 TTL：10s - 60s。
- 非强一致统计本地缓存 TTL：1s 左右。
- Redis-Cache 的业务缓存 TTL 要分散，例如 `baseTTL + random(jitter)`，避免同一批 key 同时失效。
- 热点数据过期时间设置得更长，冷门数据过期时间相对更短。
- 后台管理修改商品、拍品、直播间信息后，主动删除 Redis 缓存，并通过 Pub/Sub 通知各实例删除本地缓存。

失效模型：

```text
写 MySQL 成功
  -> 删除 Redis cache
  -> PUBLISH cache.invalidate
  -> 各 Go 实例删除本地 cache
```

如果失效通知丢失，本地 TTL 到期后也会自动修正。

### 9.5 缓存穿透、击穿和雪崩

缓存穿透指查询不存在的数据导致每次都打到 MySQL。处理策略：

- 使用 Bloom Filter 保存合法商品 ID、拍品 ID、直播间 ID；Bloom Filter 判断一定不存在时直接返回。
- MySQL 查询不存在时写入空值缓存，Redis-Cache 空值 TTL 可设置为 30s 左右，本地空值 TTL 可设置为 3s - 5s。
- 对异常 ID、非法参数、批量探测请求做接口限流。

缓存击穿指单个热点 key 过期后，大量请求同时回源。处理策略：

- 本地缓存 miss 后，对同一个 key 使用 `singleflight` 合并本机内并发回源。
- Redis-Cache miss 后，回源 MySQL 前也使用 `singleflight` 或短 TTL 互斥 key，确保同一时刻只有少量请求回源。
- 对核心热点商品可以使用逻辑过期：读到过期旧值时先返回旧值，由抢到刷新权的请求异步重建缓存。

缓存雪崩指大量 key 同时失效或 Redis-Cache 故障导致集中回源。处理策略：

- 将缓存失效时间分散开，在原有失效时间基础上增加随机值。
- 热点数据过期时间设置得更长，冷门数据过期时间设置得更短。
- 热点数据提前预热。
- Redis-Cache 故障时，对展示型接口限流或返回旧值，避免流量直接压垮 MySQL。

### 9.6 和 WebSocket 的关系

WebSocket 广播中的价格、领先者、倒计时不读本地缓存，直接使用 Lua 事件 payload。商品标题、图片、直播间标题这类展示信息可以来自本地缓存；即使短暂不一致，也不影响竞拍裁决。

## 10. Redis Stream、Kafka、MySQL 落库方案对比

### 10.1 Redis Stream -> MySQL

优点：

- 链路短。
- 延迟低。
- 组件少，运维简单。
- 适合当前系统作为第一阶段落地。

缺点：

- 多下游能力弱于 Kafka。
- 长期保留和跨系统消费能力有限。
- Redis 持久化能力弱于 Kafka。

### 10.2 Redis Stream -> Kafka -> MySQL

优点：

- Kafka 可削峰。
- 多消费者扩展能力强。
- 适合风控、数据仓库、实时大屏、审计归档等多下游。
- MySQL 慢时可以由 Kafka 缓冲。

缺点：

- 链路更长，端到端延迟更高。
- 运维和监控复杂度更高。
- 需要处理 Kafka offset、重复消息、分区顺序。

推荐演进：

```text
第一阶段：Redis Lua -> Redis Stream -> MySQL
第二阶段：Redis Lua -> Redis Stream -> Kafka -> MySQL/风控/大屏
```

Kafka 分区键必须使用 `auctionId`，保证同一拍品事件顺序。

## 11. 监控指标

建议重点监控：

- 出价接口 QPS、P95/P99 延迟、错误率。
- Redis Lua 执行耗时。
- Redis CPU、内存、instantaneous_ops_per_sec。
- 单拍品事件 seq 增长速度。
- Redis Stream pending 数量、claim 数量、DLQ 数量。
- 出价事件从 Redis 到 MySQL 的落库延迟。
- WebSocket 广播延迟：Lua seq 生成时间到客户端收到时间。
- WebSocket 单实例连接数和每拍品连接数。
- MySQL `bid_record` 批量写入耗时和冲突数量。
- 落槌 Redis 成功但 MySQL 失败的重试次数。
- 拍品启动 Redis-RT state 缺失数量、报名/保证金 Redis-RT set 补偿次数。
- 落槌 MySQL 乐观锁冲突次数、重复关闭返回已有终态次数。
- 本地缓存命中率、Redis 缓存命中率、缓存失效通知失败次数。

## 12. 落地步骤

### 阶段一：优化当前热路径

- 调整 `BidService.Place`，优先查 Redis 幂等，stream-enabled 模式下避免前置 MySQL 查询。
- 确认限流、幂等、价格校验、事件写入都在 `bid.lua` 内完成。
- 保留 MySQL 唯一键作为落库幂等兜底。

### 阶段二：WebSocket 多实例广播

- 在 Lua 中保留 `XADD Stream`，可新增 `PUBLISH` 低延迟事件。
- 每个后端实例订阅广播通道，并只广播本机连接。
- WebSocket 事件带 `seq`，客户端按 `seq` 防乱序。
- 新连接先读 Redis state 快照，再接收增量事件。

### 阶段三：落库与补偿强化

- `BidRecordWriter` 支持批量 insert。
- 保留重复消息幂等处理。
- 完善 DLQ 和 reconciliation 告警。
- 定期对账 Redis Stream seq 与 MySQL 出价记录。
- 启动拍卖采用 `WARMING_UP/STARTING -> Redis-RT 初始化 -> RUNNING` 的类 TCC/中间态流程。
- 报名/保证金采用 MySQL 权威事实 + Redis-RT 裁决索引同步写 + reconciler 补偿。
- 落槌 MySQL 更新增加 `version` 乐观锁条件，终态重复请求读取已有结果返回。
- 如引入 Kafka，关闭事件先写 Redis Stream，再由 bridge 转发 Kafka，避免 Redis 关闭成功但 Kafka 事件丢失。

### 阶段四：水平扩展

- Redis key 使用 hash tag，为 Redis Cluster 做准备。
- 按 `auctionId` 分片。
- 热门拍品独立 shard。
- 需要多下游时引入 Kafka bridge。
- 为商品、直播间、展示型拍品信息增加本地缓存和失效通知。

## 13. 结论

推荐的主线方案是：

```text
出价裁决：Redis Lua
实时广播：Lua event payload + Pub/Sub/Kafka fan-out
可靠日志：Redis Stream 或 Kafka
历史落库：异步幂等写 MySQL
最终一致：唯一键 + 重试 + DLQ + 对账补偿
```

不要在出价热路径上加分布式锁。多实例下的并发控制点应是 Redis Lua；系统吞吐通过拍品间分片、MySQL 异步化、WebSocket 事件化广播来提升。
