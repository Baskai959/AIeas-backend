# 可观测性设计方案

## 1. 目标

系统需要通过 Metrics、Trace、Logs 三类信号覆盖拍卖核心链路，做到：

- 能快速判断故障发生在哪一层：HTTP、Redis Lua、MySQL、Redis Stream/Kafka、WebSocket Gateway、Agent。
- 能追踪单次出价、落槌、订单生成、WebSocket 广播的完整链路。
- 能通过 `trace_id`、`request_id`、`auctionId`、`seq` 关联日志、指标和链路。
- 能对高并发下的延迟、积压、丢事件、慢客户端、补偿任务失败做告警。

当前项目已有基础：

- `RequestIDMiddleware` 会生成/透传 `X-Request-Id` / `X-Trace-Id`。
- HTTP 响应中包含 `trace_id`。
- `slog` 已作为结构化日志基础。
- GORM 已桥接到 `slog`，支持慢 SQL 日志。
- 配置中已有 `observability.metricsPath` 和 `slowSQLThresholdMs`。

仍需补齐：

- 真正的 Prometheus `/metrics`。
- OpenTelemetry Trace。
- Redis、Lua、Worker、WebSocket、Agent 业务指标。
- Trace context 在 Redis Stream/Kafka/WebSocket 事件中的传递。

## 2. 总体架构

```text
Go Backend / WebSocket Gateway / Agent Service
  -> Metrics: Prometheus /metrics
  -> Trace: OpenTelemetry OTLP -> Tempo / Jaeger
  -> Logs: slog JSON -> Loki / ELK

Grafana
  -> Dashboard
  -> Alert
  -> Trace drilldown
  -> Log query by trace_id
```

三类信号职责：

- Metrics：回答“现在哪里异常、影响范围多大”。
- Trace：回答“这一次请求卡在哪一步”。
- Logs：回答“为什么失败、上下文是什么”。

## 3. Trace 与日志关联字段

所有入口请求都需要生成或透传：

```text
trace_id
request_id
span_id
user_id
role
route
method
```

拍卖业务链路额外记录：

```text
auction_id
room_id
item_id
seller_id
bidder_id
order_id
idempotency_key_hash
seq
stream_id
event_type
```

注意：

- `user_id`、`auction_id` 可以进日志和 trace attribute。
- 不要把 `user_id`、`auction_id`、`request_id` 放进 Prometheus 高基数 label。
- 幂等键只记录 hash，不记录原文。
- JWT、手机号、完整 DSN、密钥不能写入日志。

## 4. Metrics 设计

### 4.1 HTTP 指标

```text
http_requests_total{method,route,status}
http_request_duration_seconds{method,route,status}
http_inflight_requests{route}
http_request_body_bytes{route}
http_response_body_bytes{route}
```

说明：

- `route` 使用模板路径，例如 `/api/v1/auctions/:id/hammer`，不要使用真实 ID。
- status 使用 HTTP 状态码或状态码段。

### 4.2 出价指标

```text
auction_bid_total{result,reason}
auction_bid_duration_seconds
auction_bid_lua_duration_seconds{script="bid"}
auction_bid_reject_total{reason}
auction_bid_duplicate_total
auction_bid_freq_limit_total
auction_bid_precheck_reject_total{reason}
auction_bid_stream_write_total{result}
```

常见 `result`：

```text
accepted
rejected
duplicate
error
```

常见 `reason`：

```text
INVALID_STATE
BLACKLIST
NOT_ENROLLED
DEPOSIT_NOT_READY
FREQ_LIMIT
BELOW_MIN_INCREMENT
REDIS_ERROR
```

### 4.3 落槌指标

```text
auction_hammer_total{result}
auction_hammer_duration_seconds
auction_hammer_lua_duration_seconds{script="hammer"}
auction_hammer_mysql_tx_duration_seconds
auction_hammer_duplicate_total
auction_hammer_optimistic_conflict_total
auction_hammer_mysql_fail_total
auction_closed_reconcile_total{result}
```

关注点：

- 自动落槌和人工落槌并发冲突数量。
- Redis 关闭成功但 MySQL 落库失败数量。
- 关闭事件补偿成功/失败数量。

### 4.4 报名与保证金指标

```text
auction_enroll_total{result}
auction_enroll_duration_seconds
auction_deposit_ready_total
auction_deposit_sync_redis_fail_total
auction_deposit_reconcile_total{result}
auction_deposit_reconcile_lag_seconds
```

用于发现 MySQL `deposit_ledger READY` 与 Redis-RT set 不一致。

### 4.5 Redis 指标

应用层补充：

```text
redis_command_duration_seconds{op}
redis_command_errors_total{op}
redis_lua_duration_seconds{script}
redis_lua_errors_total{script,error}
redis_stream_xadd_total{stream,result}
redis_stream_pending{stream,group}
redis_stream_lag_seconds{stream,group}
redis_lock_acquire_total{lock,result}
```

Redis 自身指标由 redis_exporter 采集：

```text
redis_memory_used_bytes
redis_connected_clients
redis_instantaneous_ops_per_sec
redis_keyspace_hits_total
redis_keyspace_misses_total
redis_evicted_keys_total
redis_cpu_sys_seconds_total
```

Redis-RT 重点看：

- Lua P95/P99。
- CPU。
- command error。
- stream pending。
- evicted keys 必须为 0。

Redis-Cache 重点看：

- 命中率。
- evicted keys。
- 回源 MySQL 次数。
- Bloom Filter 拦截数。

### 4.6 MySQL 指标

```text
mysql_query_duration_seconds{operation}
mysql_tx_duration_seconds{operation}
mysql_query_errors_total{operation}
mysql_slow_query_total
mysql_rows_affected{operation}
```

GORM 慢 SQL 日志继续保留，建议慢 SQL 日志中包含：

```text
trace_id
elapsed_ms
rows
sql
source
```

### 4.7 Worker / Kafka 指标

Redis Stream worker：

```text
worker_bid_record_consume_total{result}
worker_bid_record_write_duration_seconds
worker_bid_record_duplicate_total{result}
worker_bid_record_dlq_total{reason}
worker_bid_record_claim_total
worker_reconcile_total{type,result}
```

Kafka：

```text
kafka_produce_total{topic,result}
kafka_produce_duration_seconds{topic}
kafka_consume_total{topic,group,result}
kafka_consumer_lag{topic,group,partition}
```

### 4.8 WebSocket Gateway 指标

```text
ws_connections
ws_connection_total{event}
ws_subscriptions{type}
ws_heartbeat_timeout_total
ws_broadcast_duration_seconds
ws_broadcast_fanout_total
ws_broadcast_queue_length
ws_message_drop_total{reason}
ws_slow_client_disconnect_total
ws_reconnect_total
ws_event_delivery_lag_seconds{event_type}
```

关键观测：

- Gateway 是否消费事件变慢。
- 单拍品 fan-out 是否过大。
- 慢客户端是否导致队列堆积。

### 4.9 Agent 指标

```text
agent_task_total{type,status}
agent_task_duration_seconds{type}
agent_tool_call_total{tool,status}
agent_tool_call_duration_seconds{tool}
agent_rag_retrieval_total{kb,status}
agent_llm_request_total{model,status}
agent_review_decision_total{decision,risk_level}
```

Agent 不影响拍卖热路径，但需要监控任务失败率、Tool 超时、RAG 检索失败和模型调用失败。

## 5. Trace 设计

### 5.1 HTTP 出价 Trace

```text
HTTP POST /api/v1/auctions/:id/bids
  span: http.request
    attr: route, method, status, user_id, trace_id
    span: bid.place
      attr: auction_id, bidder_id, request_id_hash
      span: bid.precheck
      span: redis.idempotency.get
      span: redis.lua.bid
        attr: script=bid, accepted, reason, seq
      span: event.publish
      span: response.write
```

如果出价通过 WebSocket Gateway 转发，也需要 Gateway 入口 span：

```text
ws.bid.receive
  -> api.bid.forward
  -> bid.place
```

### 5.2 落槌 Trace

```text
POST /api/v1/auctions/:id/hammer
  span: hammer.request
    span: redis.lua.hammer
      attr: status, winner_id, price
    span: mysql.tx.close_auction
      span: mysql.update auction_lot
      span: mysql.create order_deal
      span: mysql.update deposits
    span: event.publish auction.closed
```

并发冲突时：

```text
attr:
  optimistic_conflict=true
  existing_status=CLOSED_WON
```

### 5.3 报名/保证金 Trace

```text
POST /api/v1/auctions/:id/enroll
  span: enroll.request
    span: mysql.tx.deposit_ready
    span: redis.sadd enrolled
    span: redis.sadd deposits
```

Redis 同步失败需要记录错误并触发补偿指标。

### 5.4 Worker Trace

Redis Stream / Kafka 事件需要携带 `traceparent`：

```json
{
  "traceparent": "00-...",
  "auctionId": 10001,
  "seq": 18,
  "eventType": "bid.accepted"
}
```

Worker 消费时继续 trace：

```text
stream.consume bid.accepted
  span: mysql.insert bid_record
  span: stream.ack
```

这样可以从一次 HTTP 出价追踪到异步落库。

### 5.5 WebSocket Trace

```text
pubsub.consume bid.accepted
  span: ws.event.consume
    attr: auction_id, seq, event_type
    span: ws.broadcast
      attr: subscriber_count
    span: ws.send.batch
```

不建议为每个客户端发送创建独立 span，否则数据量过大。可以对慢客户端或异常客户端采样记录。

### 5.6 Agent Trace

```text
POST /agent/tasks
  span: agent.task
    span: rag.retrieve
    span: tool.get_item
    span: tool.check_forbidden_words
    span: llm.generate
    span: save_agent_result
```

Tool 调用必须记录 tool name、latency、status，不记录敏感原文。

## 6. 日志设计

日志使用 `slog` JSON 格式。建议所有关键日志包含：

```text
timestamp
level
msg
trace_id
request_id
span_id
component
operation
auction_id
room_id
user_id
seq
error
elapsed_ms
```

关键日志点：

- HTTP 请求结束。
- Redis Lua 错误或慢调用。
- 出价 rejected 关键原因。
- 落槌事务失败。
- Worker 写 MySQL 失败。
- DLQ 写入。
- WebSocket 慢客户端断开。
- Agent Tool 调用失败。

日志示例：

```json
{
  "level": "ERROR",
  "msg": "hammer mysql transaction failed",
  "trace_id": "trc_123",
  "auction_id": 10001,
  "operation": "hammer.close",
  "elapsed_ms": 128,
  "error": "deadlock found when trying to get lock"
}
```

## 7. 健康检查

建议新增：

```text
/healthz
/readyz
/metrics
```

`/healthz`：

- 进程存活即可返回 200。

`/readyz`：

- MySQL ping。
- Redis-RT ping。
- Redis-Cache ping。
- Redis scripts loaded。
- Kafka producer/consumer 可选。

`/metrics`：

- Prometheus text format。
- 不再返回 JSON 占位。

## 8. 告警建议

核心告警：

- HTTP 5xx 比例超过阈值。
- 出价 P99 延迟超过阈值。
- Redis Lua P99 超过阈值。
- Redis-RT CPU 持续过高。
- Redis-RT evicted keys > 0。
- Stream pending 持续增长。
- BidRecord DLQ > 0。
- 落槌 MySQL 失败 > 0。
- WebSocket 重连率突增。
- WebSocket 慢客户端断开突增。
- Kafka consumer lag 持续增长。
- Agent 审查任务失败率突增。

## 9. 排障流程

### 9.1 用户反馈出价成功但页面没更新

1. 用 `request_id` / `Idempotency-Key` hash 查 HTTP trace。
2. 看 `redis.lua.bid` span 是否 accepted。
3. 看 trace 中返回的 `seq`。
4. 用 `auction_id + seq` 查事件是否写入 Stream/PubSub。
5. 查 Gateway 是否消费到该事件。
6. 看 `ws.broadcast_duration_seconds` 和发送队列长度。
7. 如果客户端有日志，确认是否因 `seq <= lastSeq` 被丢弃。

### 9.2 出价接口延迟升高

先看：

```text
http_request_duration_seconds
auction_bid_duration_seconds
auction_bid_lua_duration_seconds
redis_command_duration_seconds
```

判断：

- HTTP 慢、Lua 不慢：Go 服务 CPU、GC、连接池、限流、上游网络。
- Lua 慢：Redis-RT CPU、单拍品热点、Lua 逻辑过重。
- MySQL 慢但出价也慢：说明热路径仍有 MySQL 查询，应优化为 Redis-only。

### 9.3 落槌后订单没生成

1. 查 `hammer.request` trace。
2. 确认 `redis.lua.hammer` 是否返回 `CLOSED_WON`。
3. 查看 `mysql.tx.close_auction` 是否失败。
4. 查询 `order_deal.auction_id` 是否已有订单。
5. 查看 `auction_closed_reconcile_total` 是否补偿执行。
6. 如果进入 DLQ，按 `auction_id` 和 `trace_id` 查看错误原因。

### 9.4 报名成功但出价提示未报名

1. 查报名接口 trace，确认 MySQL `deposit_ledger READY` 是否成功。
2. 查看 Redis-RT `SADD enrolled/deposits` 是否失败。
3. 查看 `auction_deposit_sync_redis_fail_total`。
4. 等待或触发 reconciler。
5. 确认 Redis-RT set 是否补齐。

### 9.5 WebSocket 大量重连

1. 看 Gateway 连接数和重连率。
2. 看 heartbeat timeout。
3. 看 LB idle timeout 是否小于心跳间隔。
4. 看 Gateway CPU/GC。
5. 看 Pub/Sub/Kafka 消费延迟。
6. 看慢客户端队列和断开数量。

## 10. 落地步骤

### 阶段一：Metrics

- 引入 Prometheus client。
- `/metrics` 输出 Prometheus text format。
- HTTP middleware 打基础指标。
- 出价、落槌、Redis Lua、Worker、WebSocket 核心指标。

### 阶段二：Trace

- 引入 OpenTelemetry SDK。
- HTTP middleware 创建 root span。
- Redis、GORM、service 关键方法创建 child span。
- Redis Stream/Kafka 事件携带 `traceparent`。
- slog 日志注入 `trace_id`。

### 阶段三：Dashboard 与告警

- 建 Grafana dashboard：
  - API 总览。
  - 出价链路。
  - Redis-RT。
  - Worker/Kafka。
  - WebSocket Gateway。
  - Agent。
- 配置 P95/P99、错误率、积压、DLQ、重连率告警。

### 阶段四：排障工具

- 后台支持按 `trace_id` 查询链路。
- 支持按 `auctionId + seq` 查询事件状态。
- 支持按 `requestId` 查询出价结果和落库状态。
- 支持 DLQ 重放和补偿任务审计。

## 11. 标签与基数规范

Prometheus label 禁止使用：

```text
user_id
auction_id
request_id
idempotency_key
trace_id
order_id
stream_id
```

这些字段放在 trace attribute 或日志里。

Prometheus label 允许使用：

```text
route
method
status
result
reason
operation
script
component
event_type
```

## 12. 总结

可观测性链路应做到：

```text
Metrics 定位异常范围
Trace 定位单次请求卡点
Logs 解释失败原因
```

对于拍卖系统，必须重点观测：

- 出价 Redis Lua 延迟。
- 落槌事务成功率。
- Redis Stream/Kafka 积压。
- WebSocket 广播延迟和重连率。
- DB/Redis 不一致补偿次数。
- Agent 任务失败率。

所有核心事件都需要携带 `trace_id/request_id/auctionId/seq`，才能在高并发、多实例、异步链路下完成闭环排查。
