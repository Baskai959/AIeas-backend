# 后端 需求功能与开发任务文档（3 周 MVP）

> 服务定位：基于 **Go 1.22+ + CloudWeGo Hertz + Hertz WebSocket + go-redis/v9 + MySQL 8 + Redis 7** 实现的实时竞拍后端，覆盖「拍品管理 → 规则引擎 → 实时出价 → 动态排名 → anti-sniping → 落锤成交 → 保证金模拟 → 风控审计」全链路，是「实时竞拍大师」三端共用的核心。
>
> 技术栈：Go 1.22+ + CloudWeGo Hertz（推荐）+ Hertz WebSocket（`github.com/hertz-contrib/websocket`）+ go-redis/v9（含 Lua）+ MySQL 8 + Redis 7 + GORM/sqlc（推荐保留 GORM）+ JWT（golang-jwt/jwt）。

## 5. 关键技术细节

### 5.1 Redis Key 命名

| Key | 类型 | 用途 |
| --- | --- | --- |
| `auction:{id}:state` | Hash | 当前价、领先者、状态、end_ts_ms |
| `auction:{id}:bids` | ZSet | 出价排行（score = price * 1e7 - ts_ms） |
| `auction:{id}:user_bids` | Hash | 每个用户最高价 |
| `auction:{id}:idem:{rid}` | String | 幂等键，TTL 300s |
| `auction:{id}:lock:close` | String | 落锤分布式锁，TTL 10s |
| `auction:{id}:stream` | Stream | 事件流（可选审计） |

### 5.2 出价 Lua 脚本（参考技术设计文档第 4.3.2 节）

复合分值 `score = price * 1e7 - ts_ms`，确保高价优先 + 同价先到先得。

### 5.3 Go 端 Lua 调用与脚本预热

- 使用 `github.com/redis/go-redis/v9` 作为 Redis 客户端；
- 应用启动时通过 `redis.NewScript(content).Load(ctx, client)` 或 `client.ScriptLoad` 对 `bid.lua` / `hammer.lua` 执行 **SCRIPT LOAD**，将返回的 sha1 缓存到全局变量；
- 业务调用走 `EVALSHA`（即 `redis.Script.Run` / `client.EvalSha`），命中失败时回退 `EVAL` 并重新缓存 sha1；
- 多实例启动均独立预热；脚本变更通过版本化文件名 + 重新 LOAD 实现，避免运行时拼字符串。

### 5.4 幂等契约

- 客户端每次出价生成唯一 `requestId`；
- Redis `SET NX EX` 保证 5 分钟内不重复处理；
- DB `bid_record.uk_request_id` 提供持久化兜底；
- 重复请求返回上次结果（不重复推进状态）。

### 5.5 落锤一致性

- Redis 锁（`auction:{id}:lock:close`）+ Lua 原子写入；
- MySQL `auction_lot.status` 唯一约束 + 状态机校验；
- 自动到时 + 手工兜底；
- 保留价不达标 → `CLOSED_FAILED`。

### 5.6 倒计时与 anti-sniping

- 服务端权威时间，客户端仅做插值；
- 出价时若 `end_ts - now <= window`，自动延长 `extend_sec`；
- 广播 `timer.extended` 通知所有客户端。

### 7.3 WebSocket 事件

> Go 端不再使用 Socket.IO 协议，改为基于 Hertz WebSocket（`github.com/hertz-contrib/websocket`）的 WebSocket 能力 + 自定义 JSON 事件协议；事件名称与三端文档保持一致，仅传输层切换为 Hertz WebSocket。`gorilla/websocket` 或 `nhooyr/websocket` 仅作为迁移备选说明，不作为主方案。

报文格式：

```json
{ "type": "bid.place", "payload": { ... }, "requestId": "uuid", "seq": 123 }
```

- `type`：事件名（与下表一致）；
- `payload`：业务负载；
- `requestId`：客户端生成的唯一 ID（出价幂等键）；
- `seq`：服务端广播单调递增序列号，便于客户端去重与断线补偿。

房间隔离由 WebSocket Hub 管理（`map[auctionID]*Room` + `sync.RWMutex`，房间内通过 channel 串行化广播）。

| 事件 | 方向 | 说明 |
| --- | --- | --- |
| `bid.place` | C→S | 出价（带 requestId） |
| `bid.ack` | S→C | 出价 ACK |
| `bid.accepted` / `bid.rejected` | S→Room | 出价结果广播 |
| `ranking.updated` | S→Room | TopN |
| `bidder.overtaken` | S→User | 被超越定向推送 |
| `timer.tick` / `timer.extended` | S→Room | 倒计时 / 延时 |
| `auction.started` / `auction.closed` | S→Room | 状态广播 |
| `order.created` | S→Room/User | 成交单生成 |
| `announcement` | S→Room | 文案/TTS（加分） |
| `risk.event` | S→admin | 风险事件（加分） |

## 9. 风险与注意事项

1. **Lua 脚本错误高代价**：`bid.lua` / `hammer.lua` 必须有充分单测与压测，任何一个分支错误都会破坏成交一致性；
2. **时钟一致性**：服务端必须用单一时钟源（建议 Go `time.Now().UnixMilli()` 或 Redis `TIME`），不要混用；
3. **房间隔离**：必须按 `auction_id` 严格隔离 WS 房间，避免跨拍品广播；
4. **WebSocket Hub 并发安全**：Hub 维护的 `map[auctionID]*Room` 与连接集合必须使用 `sync.RWMutex` 保护，或通过单一 goroutine + channel 串行化广播写入，避免并发 map 写崩溃与广播错乱；
5. **goroutine 泄漏与 context 传递**：每条 WS 连接、每个房间循环、每个倒计时任务都需绑定 `context.Context`，连接关闭/拍品结束时 cancel；网络/Lua/HTTP 出站调用必须传 context 并设置超时，避免 goroutine 长期挂起导致内存泄漏；
6. **协议切换对前端的影响**：客户端需要从 `socket.io-client` 切换为标准 WebSocket 客户端，服务端使用 Hertz WebSocket 承载 `{type, payload, requestId, seq}` JSON 协议，需在端侧文档与联调阶段同步告知商家端、用户端、后台管理端；
7. **保证金账本边界**：模拟账本要明确状态机，避免出现「冻结成功但记录丢失」；
8. **风控误伤**：频控阈值要可配置，避免压测时把自己拦下；
9. **断线重连风暴**：Hub 需做连接限流 + 指数退避指引；
10. **Redis 持久化**：开启 AOF + 定时 RDB，避免重启丢失热状态；
11. **AI 接口异常降级**：LLM/TTS 失败必须降级为静态模板，不阻塞主链路；
12. **GC 与 GOMAXPROCS 调优（加分）**：高并发广播下注意减少短生命周期对象分配（复用 buffer、`sync.Pool`），容器化部署时显式设置 `GOMAXPROCS`（或使用 `automaxprocs`）以对齐 cgroup CPU 配额，必要时观测 `GODEBUG=gctrace=1` 调整 `GOGC`；
13. **范围失控**：所有「远期」功能不在本次实现；3 周内只交付清单内必做与计划内加分项。
