# 拍品封顶价与固定/阶梯加价规则接口变更

更新时间：2026-05-27

## 变更范围

- `POST /api/v1/auctions` 创建拍品
- `PATCH /api/v1/auctions/{id}` 编辑拍品
- WebSocket `bid.place` 出价校验

金额单位均为分。`incrementRule` 统一使用 `type` 区分固定加价和阶梯加价，不再接受没有 `type` 的旧格式。

## 创建/编辑拍品字段

| 字段 | 类型 | 创建 | 编辑 | 说明 |
| --- | --- | --- | --- | --- |
| `startPrice` | integer | 必填 | 可选 | 起拍价。 |
| `reservePrice` | integer | 必填 | 可选 | 保留价，`0` 表示无保留价。 |
| `capPrice` | integer | 可选 | 可选 | 封顶价，`0` 或不传表示无封顶价；出价达到该价格自动成交。设置时必须大于 `startPrice`。 |
| `incrementRule.type` | string | 必填 | 可选 | `fixed` 固定加价；`ladder` 阶梯加价。 |
| `incrementRule.maxBidSteps` | integer | 必填 | 可选 | 单次最多加价步数。若当前价为 `1000`、当前步长为 `100`、`maxBidSteps=5`，则本次最高可出 `1500`。 |
| `incrementRule.amount` | integer | fixed 必填 | fixed 可选 | 固定加价金额。仅 `type=fixed` 使用。 |
| `incrementRule.steps` | array | ladder 必填 | ladder 可选 | 阶梯规则。仅 `type=ladder` 使用。 |
| `antiSnipingSec` | integer | 可选 | 可选 | 反狙击触发窗口。最后 n 秒有人出价时触发延时，默认 15。 |
| `antiExtendSec` | integer | 可选 | 可选 | 反狙击延时时长，默认 30。 |
| `antiExtendMode` | string | 可选 | 可选 | `ADD` 表示结束时间增加 `antiExtendSec`；`RESET` 表示倒计时重置为 `antiExtendSec`。默认 `ADD`。 |
| `startTime` | string | 可选 | 可选 | 开拍时间。传了就是定时拍卖；不传表示未设置定时开拍。 |
| `durationSec` | integer | 可选 | 可选 | 拍卖时长。不传时，直播间上架/激活可再指定时长。 |
| `endTime` | string | 可选 | 可选 | 显式结束时间；若同时传 `startTime` 和 `durationSec`，后端按 `startTime + durationSec` 计算 `endTime`。 |

固定加价格式：

```json
{
  "type": "fixed",
  "amount": 100,
  "maxBidSteps": 5
}
```

阶梯加价格式：

```json
{
  "type": "ladder",
  "maxBidSteps": 5,
  "steps": [
    {
      "min": 0,
      "max": 10000,
      "amount": 100
    },
    {
      "min": 10000,
      "max": 50000,
      "amount": 500
    },
    {
      "min": 50000,
      "amount": 1000
    }
  ]
}
```

阶梯规则：

- `steps[0].min` 必须为 `0`
- 每档 `min` 包含，`max` 不包含
- 每档必须连续，不能有空洞或重叠
- 最后一档必须不传 `max`
- 每档 `amount` 必须大于 `0`
- `maxBidSteps` 对固定加价和阶梯加价都生效

```json

  "auctionType": "ENGLISH",
  "startPrice": 1000,
  "reservePrice": 5000,
  "capPrice": 12000,
  "depositAmount": 2000,
  "incrementRule": {
    "type": "fixed",
    "amount": 100,
    "maxBidSteps": 5
  },
  "antiSnipingSec": 15,
  "antiExtendSec": 30,
  "antiExtendMode": "RESET",
  "durationSec": 172800,

  "startTime": "2026-06-01T05:57:54.000Z"

```

## 编辑拍品示例

```json
{
  "capPrice": 12000,
  "incrementRule": {
    "type": "ladder",
    "maxBidSteps": 8,
    "steps": [
      {
        "min": 0,
        "max": 10000,
        "amount": 100
      },
      {
        "min": 10000,
        "amount": 500
      }
    ]
  }
}
```

运行中、延时中、终态拍品不能修改价格规则字段。创建/编辑接口传入的 `status` 只能是 `DRAFT` 或 `PENDING_AUDIT`。

## 出价校验

WebSocket `bid.place` 的 `payload.price` 必须同时满足：

- `price > startPrice`
- 如果设置了 `capPrice`，则 `price <= capPrice`
- 当前步长由当前 Redis 最高价匹配 `incrementRule` 得到：固定加价取 `amount`，阶梯加价取当前价所在档位的 `amount`
- 普通出价必须满足 `(price - currentPrice) % 当前步长 == 0`
- 如果 `price == capPrice`，允许封顶成交价不按步长对齐
- 普通出价需要 `price >= currentPrice + 当前步长`；封顶出价需要 `price > currentPrice`
- 如果设置了 `capPrice`，则 `price <= min(capPrice, currentPrice + 当前步长 * maxBidSteps)`；否则 `price <= currentPrice + 当前步长 * maxBidSteps`

示例：`startPrice=1000`、`currentPrice=1000`、`capPrice=1950`、`amount=100`、`maxBidSteps=10`：

- `1050` 拒绝：不是当前价加价步长
- `1100` 接受：当前价 + 1 步
- `1500` 接受：当前价 + 5 步
- `1950` 接受并自动成交：封顶价不要求步长对齐

达到 `capPrice` 时，Lua 会原子更新 Redis 拍卖状态为 `CLOSED_WON`，后端随后补齐拍品、订单、保证金等数据库落库。
