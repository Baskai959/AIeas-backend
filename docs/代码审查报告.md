## 3. 模块审查详情

### 3.1 Lua 脚本 (`scripts/lua/`)

**bid.lua** — ✅ 正确
- 状态检查 `status ~= "RUNNING" and status ~= "EXTENDED"` 正确覆盖 EXTENDED 状态
- 阶梯加价 `min_increment_for_price` 逻辑与 Go 域函数 `MinIncrementForPrice` 一致（均使用 `[min, max)` 区间）
- 同价先到先得：`tie_bid` 条件使排行榜不替换已有领先者
- 频控使用 `INCR + PEXPIRE` 滑动窗口近似实现，可接受

**hammer.lua** — ⚠️ 存在 Issue #1
- 仅检查 ranking ZSET 是否有 member 来决定 `CLOSED_WON` vs `CLOSED_FAILED`
- 未传入/使用 `reserve_price`
- Go 层 `hammer.go:102` 做了保留价兜底，但 Redis 状态会短暂标记为 CLOSED_WON

### 3.2 域模型 (`internal/domain/`)

- 状态机转换表 `CanTransitionAuction` 完整覆盖 AGENTS.md 定义的所有路径
- `AuctionStatus.Terminal()` 正确包含 `ClosedWon`, `ClosedFailed`, `Settled`
- 增价规则验证 `ValidateIncrementRule` 正确强制最后一个 step 无 Max 上界
- **缺失**: 无 `CanAcceptBid()` 方法（虽然在 summary 中提及，实际代码中 `AuctionStatus` 上没有该方法，状态判断散布在各调用方）

### 3.3 服务层 (`internal/service/`)

**BidService** — 良好
- 多层幂等：先查 MySQL `bid_record`，再查 Redis idempotency key，再 Lua 内部幂等
- 出价前做一次 Go 侧的 `belowMinimumIncrement` 快速拒绝，减少 Lua 调用

**HammerService** — 良好（上轮 summary 中的 "sellerID 未赋值" bug 已在当前代码中修复，`current.SellerID` 从 auction 读取正确传入 `OrderDeal`）

**AuctionService** — 注意 Issue #3
- `Start` 方法同时完成状态转换 + Redis 初始化 + 广播 + timer 调度，职责密集但可接受
- `Cancel` 直接跳转到 `ClosedFailed`，`CanTransitionAuction` 允许从所有非终态到 `ClosedFailed`

**TimerScheduler** — 功能正确，注意 Issue #4
- 到期后以 `system`/`admin` 角色调用 `Hammer`，权限校验通过
- 缺少重启恢复（已知 gap，文档 `项目完成情况与未完成任务.md` 已记录）

**BidRecordWriter** — 设计完善（亮点 E）

### 3.4 仓储层 (`internal/repository/`)

- MySQL 实现统一使用 `dbFor(ctx)` 支持事务上下文传递
- Memory 实现完整覆盖所有接口方法，测试可直接使用
